## 1. デプロイ手順 [必須]

### 1.1 前提条件
- ADF v1.2.0/1/2/3/4 完了 (or 並行進行、本 spec は独立)
- macOS dev fleet (Hook 前提)
- GitHub Actions 利用可能

### 1.2 手順
1. `git pull origin main`
2. `framework hook init` (v1.2.1 既、本 spec で pre-commit-evidence 追加)
3. `.github/workflows/evidence-check.yml` を repo に配置
4. `.github/PULL_REQUEST_TEMPLATE.md` を repo に配置 (default §Evidence)
5. `ADF_EVIDENCE_STRICT=0` (bootstrap、warn only) で start、dogfood 1 週間後 strict=1 に切替
6. PR で動作確認

### 1.3 デプロイ後確認
- 任意 PR で §Evidence 不在 → CI exit 2 (strict=1 時) or warn (strict=0)
- `framework evidence verify <pr>` で claim 検証可能

## 2. ロールバック手順 [必須]

### 2.1 条件
- evidence-check.yml が false positive 連発 (dev 速度阻害)
- pre-commit hook が誤 block

### 2.2 手順
1. `ADF_EVIDENCE_STRICT=0` で全環境を warn only mode に切替
2. evidence-check.yml を `*.disabled` 名にして CI から除外
3. pre-commit-evidence-check.sh を `--no-verify` で個別 bypass (audit log 記録)
4. PR で revert reflect

## 3. 監視項目 [必須]

| メトリクス | 正常範囲 | アラート | 通知先 |
|---|---|---|---|
| `adf_evidence_block_total` (false block) | < 5/week (strict 期) | > 50/week | #dev-adf |
| `adf_evidence_verify_failure_total` | < 3/week | > 10/week | #dev-adf |
| `adf_evidence_bypass_total` (`--no-verify`) | 0/week | > 0 (audit + CTO mention) | CTO + CEO |

## 4. SLO [必須]

| SLI | 目標 |
|---|---|
| §Evidence 強制率 (strict mode) | 100% |
| evidence verify false positive | < 5% |
| `[検証済]` overclaim 検出 | dogfood 期 ≥ 3 件 |

## 5. 障害対応 Runbook [必須、3症状以上]

### 5.1 症状: §Evidence 不在 PR が頻発
- 一次対応: PR template 配置確認、dev fleet で template review
- エスカレーション: 1 日 5+ 件で lead-bot mention
- 再発防止: PR template に visible warning 追加

### 5.2 症状: evidence verify false positive (file 引用 broken 誤検出)
- 一次対応: parser regex の verify、line range 計算ミス確認
- エスカレーション: 同症状 3 件で auditor mention
- 再発防止: parser unit test 拡充

### 5.3 症状: pre-commit hook が dev 体験悪化
- 一次対応: dev に `--no-verify` 緊急 bypass を許可 (audit log 必須)
- エスカレーション: 1 週間 5+ bypass で CEO escalate (本 spec の運用見直し)
- 再発防止: hook の sensitivity tuning、warn-only mode 導入検討

### 5.4 症状: URL 404 で false fail
- 一次対応: retry 3 後 warn のみ (block ではない、§9 fail-open spec 通り)
- 再発防止: link rot 監視を別 spec で扱う

## 6. 定期メンテナンス
- 月次: evidence verify metric review、false positive rate 確認
- 四半期: §Evidence section の dev feedback 集約、format 改訂検討
