# OPS: ADF v1.2.1 — Claude Code Hooks 基盤統合

## 0. 対応する SPEC / IMPL [必須]

- SPEC-DOC4L-008 (`docs/spec/v1.2.1-hooks.md`)
- IMPL-DOC4L-008 (`docs/impl/v1.2.1-hooks.md`)

## 1. デプロイ手順 [必須]

### 1.1 前提条件

- ADF v1.2.0 merged 済 (4 層体系 + 原則 0)
- repo に `.framework/` directory 存在 (`framework init` 済)
- Claude Code v0.x (公式 hooks supported) install 済
- haishin-puls-hub が dogfooding 環境として ready

### 1.2 手順

1. `git pull origin main` (本 PR merge 後の main)
2. 対象 repo で `framework hook init` 実行
3. `.claude/settings.json` の差分を `git diff` で確認、不要 hook は手動除去
4. `framework hook validate` で公式仕様適合確認
5. `framework hook test` で各 hook の trigger / block / pass を dry-run 検証
6. `git add .claude/settings.json .claude/scripts/` + commit + PR
7. PR が merge された時点で fleet 全 bot session に hook が反映 (各 bot は SessionStart 時に再読込)

### 1.3 デプロイ後確認

- `framework hook list` で active hook 4 種を確認
- 任意の `Edit` / `Write` 操作 → `.framework/hook-log/{date}.jsonl` に PostToolUse entry が記録されること
- dummy `gh pr merge --admin` 試行 → exit 2 で block されること
- bot completion 試行 → Stop hook が trigger されること

## 2. ロールバック手順 [必須]

### 2.1 ロールバック条件

- hook script が bot session を完全 block (exit 2 連発で進行不能)
- hook validate が CI で頻繁 false fail
- `framework hook init` 後の重大 regression 発覚

### 2.2 手順

1. `framework hook init --rollback` (= 直近 `.bak` から復元)
2. または `.claude/settings.json` を git revert + push、PR で main に戻す
3. fleet bot に SessionStart 再起動を促す (auto-restart の場合は次 session で反映)
4. `.framework/hook-log/error/{date}.log` を保全し、root cause 分析

## 5. 障害対応 Runbook [必須、3症状以上]

### 5.1 症状: bot session が Stop hook で完了できない (false block)

- 一次対応:
  1. `.framework/hook-log/{date}.jsonl` で reason を確認
  2. test suite が flaky なら該当 test を skip + follow-up issue
  3. `ADF_HOOK_DRY_RUN=1` で hook を log only mode に切替 (緊急時)
- エスカレーション: 同症状 3 件以上 / 24h で CTO + lead-ama mention
- 再発防止: hook script に flaky test 検知 + retry を追加 (別 PR)

### 5.2 症状: PreToolUse hook が誤 block (legitimate command を block)

- 一次対応:
  1. block reason を hook-log で確認
  2. pattern が過剰汎化していないか `block-dangerous.sh` を review
  3. urgent な場合は `ADF_HOOK_DRY_RUN=1` で log only に切替
- エスカレーション: CEO escalate (governance gate に関わる)
- 再発防止: pattern 修正 PR、VERIFY 異常系 §3 に該当 case 追加

### 5.3 症状: hook script crash 連発

- 一次対応:
  1. `.framework/hook-log/error/{date}.log` を確認
  2. dependency (prettier / eslint / tsc) の install 状態 check
  3. crash 連発時は `framework hook init --rollback` で前版復元
- エスカレーション: 1h 5+ crash で CTO mention
- 再発防止: hook script に dependency 不在時の graceful skip 追加 (warn のみ、block しない)

### 5.4 症状: SessionStart hook で context 注入失敗

- 一次対応:
  1. `inject-spec-context.sh` の log を確認
  2. recent spec 抽出 logic が repo 配下 path を見ているか
  3. fallback として bot に手動で `cat docs/spec/...` を促す
- エスカレーション: lead-ama mention (spec 同期に支障)
- 再発防止: hook 内 fallback (recent spec 不在時は空 additionalContext)

## 9. 制御機構の使い分け原則

> CTO directive `7212f21d` (CEO `a2dc8179`) per、本 OPS doc に統合。

ADF 原則 0 (script 制御絶対) の実装形態は 2 種類:

- **script 制御 (広義)**: 外部 process (daemon / cron / launchd / pg trigger)
- **Boris 式 Hook**: Claude Code session 内 hook (PreToolUse / Stop / SessionStart / PostToolUse)

**集合関係**: Boris 式 Hook ⊂ script 制御 (広義)

**採択原則**: **script 制御 first、Hook は不可避 4 case のみ fallback**

| Hook 不可避 case | event | 例 |
|---|---|---|
| tool 呼出 BLOCK | PreToolUse | F4-4 `block-dangerous.sh` |
| LLM context 注入 | UserPromptSubmit / SessionStart | F4-3 `inject-spec-context.sh` |
| session 起動時 state 復元 | SessionStart | F4-3 (同上) / agent-memory boot |
| tool 実行直後の検証 | PostToolUse | F4-1 `post-edit-verify.sh` |

これ以外は **すべて script 制御 (daemon / cron / pg trigger 等) で実装**。
