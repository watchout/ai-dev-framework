# OPS: Shirube Merge Authority Gate

## 0. Corresponding SPEC
`docs/spec/v1.2.0-merge-authority-gate.md` / SPEC-DOC4L-019.

## 1. Required Repository Setup
For every governed Shirube repository:

1. Configure `.framework/config.json` authority roles.
2. Ensure authority roles resolve to GitHub users or teams.
3. Install `merge-authority.yml`.
4. Enable branch protection on protected branches.
5. Require the `shirube merge-authority` status check.
6. Enable stale approval dismissal where available.

## 2. Operator Flow

### Before merge
```bash
shirube merge doctor <pr-number>
```

If doctor blocks, do not merge. Obtain missing current-head approval evidence.

### Merge
Preferred:
```bash
shirube merge <pr-number>
```

Allowed only after required check passed:
```bash
gh pr merge <pr-number>
```

Connector/web merges are allowed only when GitHub branch protection shows `shirube merge-authority` passing for the current head.

## 3. Incident Handling

### Missing required check
If a governed PR can merge without `shirube merge-authority`:
- mark the repository governance state as degraded;
- create an audit issue;
- block public MCP-quality claims;
- add the required status check before the next governed merge.

### Wrong actor approval
If approval was accepted from an unbound actor:
- block merge;
- inspect role identity mapping;
- add a regression fixture.

### Stale approval accepted
If approval from an old head passes:
- block merge;
- verify GitHub review commit id handling;
- enable stale approval dismissal;
- add a stale-head regression.

### Dismissed or superseded approval accepted
If a dismissed approval, or an actor's earlier approval followed by a later `CHANGES_REQUESTED`, passes:
- block merge;
- verify effective latest non-dismissed review-state reduction;
- add a dismissed/superseded-review regression.

## 4. Downstream Adoption Rule
Totonoe, kodama, and other downstream repositories may continue design/prep work before #184 is implemented.

They must not claim strict public MCP-quality merge governance until:
- `shirube merge-authority` is installed;
- branch protection requires the check;
- role bindings resolve to GitHub identities;
- current-head approval evidence is verified.

## 5. Rollback
If the gate causes false blocks:
- do not disable branch protection silently;
- open an incident issue;
- temporarily mark repository as `standard` or `minimal` for non-release work;
- require CTO/L3 approval for any emergency merge.

## 6. Public Release Note
v1.2.0 public MCP-quality completion requires #184 MVP:
- deterministic current-head L3/release/human approval validation when required by route, workflow policy, or audit level;
- required status check;
- branch protection documentation;
- bypass regression.
