### 1.2 手順

[推測: IMPL §1 配置図 + §3 シーケンス を運用化]:

1. `git pull origin main` (本 PR merge 後の main)
2. `framework hook init` を再実行 (settings.json に label-check.sh が追加される)
3. `.github/workflows/post-merge-verify.yml` を repo に配置 (template から copy or 手動配置)

5. `.framework/config.json` に Plan Mode opt-in flag を設定 (`ADF_PLAN_REQUIRED=1`、初期は推奨)
6. `framework dispatch task-1 bot-x --mode=plan` 試行で Plan ファイル生成を確認
7. `framework verdict log <bot> spec-decision "test"` 試行で書込みを確認
8. PR 起票 → CTO L3 sanity → CEO L4 → merge

## 2. ロールバック手順 [必須]

### 2.1 ロールバック条件

[推測: SPEC §9 リスクから導出]:
- label check の false positive rate > 20% (L1 review が滞る)
- post-merge verify が長時間化 (10 分超が連続) で developer feedback 不能
- Plan Mode 必須化で legitimate hotfix が頻繁に block

### 2.2 手順

1. `.framework/config.json` で `ADF_PLAN_REQUIRED=0` に切替 (Plan Mode を opt-out)
2. `ADF_LABEL_CHECK_DRY_RUN=1` env を bot fleet に展開 (label check が log only)
3. `.github/workflows/post-merge-verify.yml` を一時 disable (workflow rename `*.yml.disabled`)
4. `framework hook init --rollback` で settings.json を v1.2.1 状態に戻す
5. PR で revert 反映、main に push、fleet bot は SessionStart 再起動で反映

## 5. 障害対応 Runbook [必須、3症状以上]

### 5.1 症状: label check が false positive 連発、bot 進行不能

[推測: SPEC §9 リスク緩和を運用化]:
- 一次対応:
  1. `.framework/hook-log/{date}.jsonl` で reason / pattern を確認
  2. 過剰汎化なら `ADF_LABEL_CHECK_DRY_RUN=1` で log only 化 (緊急時)
  3. label-check.sh の regex 修正 PR 起票
- エスカレーション: 1h で 10+ false block → lead-ama + CTO mention
- 再発防止: VERIFY 異常系に false positive case 追加、regex unit test 拡充

### 5.2 症状: citation 検証で legitimate citation が誤 block

- 一次対応:
  1. block log で file:line 抽出文と実 file content を diff
  2. 行末空白 / 改行差分等の正規化漏れなら fix PR
  3. urgent には `ADF_LABEL_CHECK_DRY_RUN=1` で skip
- エスカレーション: 同 bot で 3 件以上 → auditor mention
- 再発防止: 正規化処理を `lib/label/citation.ts` に集約 + unit test

### 5.3 症状: post-merge verify が timeout (10 分超)

- 一次対応:
  1. GH Actions log で test suite 個別の duration 確認
  2. 重い test を `--shard` で並列化、または skip + follow-up issue
  3. timeout 設定を 15 分に拡大 (一時、CTO 承認下)
- エスカレーション: 連続 3 commit で timeout → CTO + CEO mention
- 再発防止: test suite 軽量化 PR、heavy test を nightly に分離

### 5.4 症状: Plan Mode 必須で hotfix が block

- 一次対応:
  1. `ADF_PLAN_BYPASS=1` で 1 回限り override (audit log + Discord 自動 post)
  2. CTO に bypass 使用を即時報告
  3. hotfix 後すぐに Plan ファイルを retroactive 作成 (post-mortem 文書化)
- エスカレーション: 月 5+ bypass → SPEC 見直し (Plan Mode の hard 必須化を再評価)
- 再発防止: hotfix 用 Plan template を別途用意、`framework dispatch --mode=hotfix` を別 spec で検討

### 5.5 症状: verdict log が膨大化、disk 圧迫

- 一次対応:
  1. `.framework/verdicts/<bot>/` の容量確認
  2. `ADF_VERDICT_RETENTION_DAYS=30` (短縮) で運用、archive を別 path
- エスカレーション: 1 GB 超 → CTO mention
- 再発防止: archive 自動化 cron (本 spec 範囲外、別 PR)

## 9. 制御機構の使い分け原則

> [文献確認: CTO directive `7212f21d` (CEO `a2dc8179`)] 本 OPS doc に統合。

ADF 原則 0 (script 制御絶対) の実装形態は 2 種類:

- **script 制御 (広義)**: 外部 process (daemon / cron / launchd / pg trigger / GH Actions)
- **Boris 式 Hook**: Claude Code session 内 hook (PreToolUse / Stop / SessionStart / PostToolUse)

**集合関係**: Boris 式 Hook ⊂ script 制御 (広義)

**採択原則**: **script 制御 first、Hook は不可避 4 case のみ fallback**

| Hook 不可避 case | event | 例 (v1.2.x で導入) |
|---|---|---|
| tool 呼出 BLOCK | PreToolUse | v1.2.1 F4-4 `block-dangerous.sh` |
| LLM context 注入 | UserPromptSubmit / SessionStart | v1.2.1 F4-3 `inject-spec-context.sh` / v1.2.2 F1-1 拡張 |
| session 起動時 state 復元 | SessionStart | v1.2.2 F5 verdict auto-load (新 session) |
| tool 実行直後の検証 | PostToolUse / Stop | v1.2.1 F4-1 `post-edit-verify.sh` / v1.2.2 F2-F3 `label-check.sh` |

これ以外は **すべて script 制御 (daemon / cron / GH Actions / pg trigger 等) で実装**。本 spec の F4 post-merge verify は GH Actions で **script 制御層**、F2/F3 label/citation は Stop hook で **case 4 = tool 実行後検証 (= bot 完了直前) の hook 層**。
