# OPS: Shirube Gate Engine

## 0. Corresponding SPEC
`docs/spec/v1.2.0-gate-engine.md` / SPEC-DOC4L-020.

## 1. Adoption Modes

### minimal
Use for local-only or early projects.

Operational rule:
- observe/warn only;
- no GitHub requirement;
- no public MCP-quality claim.

### standard
Use for remote task queues and team workflows.

Operational rule:
- GitHub may be task/evidence SSOT when configured;
- remote publish requires role/policy readiness;
- local draft work remains possible during migration.

### strict
Use for public-grade dogfood and production/high-risk repos.

Operational rule:
- concrete roles required;
- hearing/design/review/release gates require evidence;
- merge authority and branch protection are required for release claims.

## 2. Operator Flow
1. Run existing Shirube workflow normally.
2. Use `shirube workflow status --json` to inspect phase/evidence.
3. Resolve role readiness through `shirube roles doctor` and `workflow-config.ts`.
4. Use `shirube gate design --strict` for automation-related design validation.
5. Use `shirube workflow doctor --json` for diagnosis; `doctor` is not an enforcement command.
6. Use `shirube workflow check --action <action> --json` only as an explicit local check until a later PR maps actions to enforcement surfaces.
7. Do not connect `workflow check` to `run`, hooks, CI, GitHub Checks, issue/PR generation, or design generation until PR A/B/C have landed and the action mapping has been audited.
8. For any PR contributing to a phase exit claim, record post-merge verification evidence per `docs/specs/postmerge-001.md`.

## 3. Downstream Guidance

### Kodama
- Controlled update is allowed.
- Do not claim strict/public MCP-quality until role bindings, validation PR, merge-authority required check, and stale approval dismissal are active.
- Use workflow observability after PR C before broader strict adoption.

### Totonoe
- Start with `/design`.
- Do not mix AI Concierge assumptions into Totonoe specs.
- Wait for PR B/C-level observability before formal strict bootstrap.

### Wasurezu / agent-mem
- Treat memory/restart context as integration evidence, not phase authority.
- Apply strict Gate Engine only after #204 validator hardening.

## 4. Incident Handling
| Incident | Response |
|----------|----------|
| local project blocked by GitHub dependency | treat as bug; minimal/local mode must not require GitHub |
| legacy discover/generate breaks | revert enforcement change; PR A/B/C must be compatibility-preserving |
| GitHub issue treated as hearing completion | block rollout; issue intake is not hearing completion |
| role authority duplicated | revert duplicate model; use `workflow-config.ts` / `merge-authority.ts` |
| hook becomes state authority | revert/harden; hooks are not canonical workflow state |
| LLM output used as approval | block; require deterministic evidence |
| phase-critical PR lacks post-merge verification | block phase closure until a post-merge record exists or L3 explicitly re-dispositions the gap |

## 5. Rollback
PR A is docs-only and rollback is a doc revert.

PR B/C rollback must preserve existing local artifacts:
- do not delete discovery sessions;
- do not rewrite current-session without migration evidence;
- do not remove `.framework/config.json`;
- do not mutate GitHub Issues/PRs in read-only slices.

## 6. Privacy And Projection
- Local-only evidence stays local unless explicitly published.
- Redact secrets, tokens, local paths, and sensitive PII before GitHub/public projection.
- Do not store private reasoning traces as gate evidence.
- Evidence should carry `privacy_scope`.

## 7. Broad Rollout Readiness
Before broad rollout to Kodama/Totonoe/Wasurezu:
- #195 is merged and #204 hardening is complete;
- PR A/B/C are merged;
- workflow status/check/explain outputs are audited;
- local-only fixture passes;
- strict fixture passes;
- Shirube dogfood report exists.
