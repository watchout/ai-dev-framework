# OPS: Shirube Gate Engine

## 0. Corresponding SPEC
`docs/spec/v1.2.0-gate-engine.md` / SPEC-DOC4L-020.

## 1. Adoption Modes

### minimal
Use for local-only or early projects.

Operational rule:
- observe/warn only;
- no GitHub requirement;
- no public MCP-quality claim.

### standard
Use for remote task queues and team workflows.

Operational rule:
- GitHub may be task/evidence SSOT when configured;
- Goal Contract, phase plan, feature catalog, task DAG, Context Pack, provider/tool policy, and AI Change Record are visible as workflow readiness evidence;
- remote publish requires role/policy readiness;
- local draft work remains possible during migration.

### strict
Use for public-grade dogfood and production/high-risk repos.

Operational rule:
- concrete roles required;
- Goal Contract, hearing, phase plan, feature catalog, task DAG, SPEC/IMPL readiness, Context Pack, provider/tool policy, pre-impl audit, implementation audit, AI Change Record, GitHub Check projection, review, release, post-merge verify, and goal progress gates require evidence;
- merge authority and branch protection are required for release claims.

## 2. Operator Flow
1. Choose adoption mode: greenfield, retrofit, or managed resume/update.
2. Resolve role readiness through `shirube roles doctor` and `workflow-config.ts`.
3. Draft and approve the Goal Contract before claiming V0/V1 direction.
4. Decompose the approved goal into phase plan, P0/P1 feature catalog, and task DAG.
5. Create task-level SPEC/IMPL/VERIFY/OPS docs with trace links to goal sufficient conditions.
6. Build a task Context Pack and declare provider/tool policy before implementation starts.
7. Run pre-impl audit before implementation starts.
8. Implement and run implementation audit, CI, test audit, and trace verification.
9. Generate an AI Change Record for the PR/change set.
10. Project gate/evidence status to GitHub Check / PR comment for remote workflows.
11. Evaluate merge authority.
12. Run post-merge verification and update goal progress evidence.
13. Use `shirube workflow status --json` and `shirube workflow doctor --json` to inspect phase/evidence. `doctor` is diagnostic only.
14. Use `shirube workflow check --action <action> --json` only as an explicit local check until a later PR maps actions to enforcement surfaces.
15. Do not connect `workflow check` to `run`, hooks, CI, GitHub Checks, issue/PR generation, or design generation until PR A/B/C have landed and the action mapping has been audited.

## 3. Downstream Guidance

## 4. Incident Handling
| Incident | Response |
|----------|----------|
| local project blocked by GitHub dependency | treat as bug; minimal/local mode must not require GitHub |
| legacy discover/generate breaks | revert enforcement change; PR A/B/C must be compatibility-preserving |
| GitHub issue treated as hearing completion | block rollout; issue intake is not hearing completion |
| GitHub issue treated as Goal Contract approval | block rollout; goal approval requires configured human/governance confirmation |
| feature/task/PR has no goal trace | block strict rollout; report uncovered Goal Contract condition |
| AI worker starts with generic prompt only | block strict rollout; require Context Pack id/hash |
| provider/tool policy missing or implicit | block strict rollout; require explicit provider/tool policy evidence |
| AI Change Record missing before remote release claim | block; PR is not enterprise-ready without change evidence |
| GitHub Check projection leaks sensitive local context | revoke projection and fix redaction before broad rollout |
| local implementation success reported as goal completion without post-merge evidence | block; distinguish implementation success from goal progress |
| role authority duplicated | revert duplicate model; use `workflow-config.ts` / `merge-authority.ts` |
| hook becomes state authority | revert/harden; hooks are not canonical workflow state |
| LLM output used as approval | block; require deterministic evidence |

## 5. Rollback
PR A is docs-only and rollback is a doc revert.

PR B/C rollback must preserve existing local artifacts:
- do not delete discovery sessions;
- do not rewrite current-session without migration evidence;
- do not remove `.framework/config.json`;
- do not mutate GitHub Issues/PRs in read-only slices.

## 6. Privacy And Projection
- Local-only evidence stays local unless explicitly published.
- Redact secrets, tokens, local paths, and sensitive PII before GitHub/public projection.
- Do not store private reasoning traces as gate evidence.
- Evidence should carry `privacy_scope`.

## 7. Broad Rollout Readiness
Before broad rollout to Kodama/Totonoe/Wasurezu:
- #195 is merged and #204 hardening is complete;
- PR A/B/C are merged;
- Goal Contract, phase plan, feature catalog, task DAG, SPEC/IMPL readiness, Context Pack, provider/tool policy, AI Change Record, GitHub Check projection, audit, release, post-merge verify, and goal progress checks are specified and dogfooded;
- workflow status/check/explain outputs are audited;
- local-only fixture passes;
- strict fixture passes;
- Shirube dogfood report exists.
