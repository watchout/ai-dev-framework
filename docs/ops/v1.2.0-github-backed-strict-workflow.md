# OPS: Shirube v1.2.0 Completion — GitHub-Backed Strict Workflow

## 0. 対応する SPEC
SPEC-DOC4L-018。

## 1. Operator policy
Use mode by work type:

| Work type | Mode | GitHub required |
|---|---|---|
| public MCP-quality work | strict | yes |
| normal multi-agent repo work | standard | yes |
| local draft / personal exploration | minimal | no |

Local cache files are not the public-quality SSOT.

## 2. Applying Shirube to a repo
Choose the lifecycle path before running commands.

| State | When | Operator flow |
|---|---|---|
| S1 greenfield | Shirube is not installed and the product/repo is new | `shirube init <name>` → configure roles → `shirube start --feature <id> --audit-level strict` |
| S2 midstream adoption | Existing development repo, Shirube not installed | `shirube retrofit --scan-only` → review report → `shirube retrofit --generate` → configure roles → `shirube start --feature <id> --audit-level strict` |
| S3 managed resume/update | Shirube already installed | `shirube update` if needed → `shirube roles doctor` → `shirube start --resume` or `shirube start --force --feature <id>` |

Common strict prerequisites:

1. Ensure repository is on GitHub and `gh auth status` passes.
2. Configure all concrete roles, including `l3_governance_owner`.
3. Run `shirube roles doctor`.
4. Run queue doctor / sync once implemented.
5. Run the state-appropriate `shirube start ...` command.
6. Record the session boundary in the PR/issue trail.

`init`, `retrofit`, and `update` are application/update operations only. They do not start framework-led development until `start` creates or resumes `.framework/current-session.json`.

## 3. Design-change intake during active work
When a user, reviewer, auditor, or external instruction introduces a new design decision:

1. Classify the impact: SPEC / IMPL / VERIFY / OPS.
2. If SPEC impact exists, stop implementation.
3. Update affected docs.
4. Run `/gate-design`.
5. Record authority decision.
6. Resume implementation only after the gate passes.

Small changes may continue without full redesign only when they do not affect product behavior, public contract, role authority, security/trust boundary, task queue semantics, or release policy.

## 4. GitHub task queue operation
- Add tasks as GitHub Issues with canonical seq metadata.
- Insert tasks by gap allocation when possible.
- Run resequence when gaps are exhausted.
- Do not let autonomous agents run when sequence metadata is duplicate or missing.
- Keep `plan.json` as cache only.

## 5. Rollback
- If strict start blocks due to GitHub readiness, use `minimal` only for local draft and do not claim public quality.
- If update changes operator-owned config, restore from git and treat as release blocker.
- If queue metadata drifts, stop autonomous execution and run queue doctor/resequence after review.
