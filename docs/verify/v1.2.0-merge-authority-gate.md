# VERIFY: Shirube Merge Authority Gate

## 0. Corresponding SPEC
`docs/spec/v1.2.0-merge-authority-gate.md` / SPEC-DOC4L-019.

## 1. Required Unit Tests

| Case | Expected |
|------|----------|
| missing L3 approval | BLOCK |
| wrong actor approval | BLOCK |
| producer actor approval | BLOCK |
| placeholder `l3_governance_owner` | BLOCK |
| non-GitHub authority binding with no mapping | BLOCK |
| stale approval on old head SHA | BLOCK |
| dismissed approval | BLOCK |
| latest current-head review is CHANGES_REQUESTED after earlier APPROVED | BLOCK |
| label-only approval | BLOCK |
| current-head approval from configured L3 actor | PASS |
| `route:ceo-approval` with only L3 approval | BLOCK |
| `approval_required` without `human_approver` approval | BLOCK |
| `auto_publish` without required L3 approval | BLOCK |
| malformed `.framework/config.json` | BLOCK |

## 2. Required Integration Tests

### 2.1 GitHub adapter fixture tests
Use recorded or mocked GitHub API fixtures for:
- current head SHA;
- reviews on current head;
- reviews on stale head;
- dismissed reviews;
- later `CHANGES_REQUESTED` after earlier `APPROVED`;
- team membership;
- draft PR;
- labels.

### 2.2 Workflow test
Run `merge-authority.yml` in a fixture repository or mocked Action environment.

Assertions:
- check exits non-zero on missing evidence;
- check exits zero on valid evidence;
- JSON output includes required roles, evidence source, and head SHA.
- workflow triggers include `pull_request_review` submitted, edited, and dismissed events.

### 2.3 CLI tests
- `shirube merge doctor` is read-only and exits non-zero on block.
- `shirube merge` refuses to merge when doctor blocks.
- `shirube merge` calls merge only after doctor passes.

## 3. Bypass Regression
Policy-level regression must prove that direct merge paths are not trusted:

- `gh pr merge` is documented as allowed only after `shirube merge-authority` passed on current head.
- connector merge is documented as allowed only after `shirube merge-authority` passed on current head.
- branch protection setup includes required status check.
- existing auto-merge workflows cannot merge based only on labels such as `audit-passed`.

## 4. Manual Verification Checklist
- Enable branch protection on a test branch.
- Require `shirube merge-authority`.
- Enable stale approval dismissal.
- Open a PR with no approval: merge blocked.
- Approve from wrong actor: merge blocked.
- Approve from configured L3 actor: check passes.
- Request changes after approval: check blocks.
- Dismiss approval: check blocks.
- Push a new commit: check blocks until new approval.
- Add `route:ceo-approval`: check blocks until release/human evidence is present.

## 5. Exit Criteria
The feature is not complete until:
- all required unit tests pass;
- workflow check is required by branch protection in the Shirube repo;
- #183 violation class cannot be reproduced;
- downstream docs for Totonoe/kodama state that strict adoption requires the same required check.
