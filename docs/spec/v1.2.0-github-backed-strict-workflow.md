# SPEC: Shirube v1.2.0 Completion — GitHub-Backed Strict Workflow

## 0. メタ
- 作成日: 2026-05-21
- 起点 Issue: #167, #178
- 関連 PR: #165, #176
- 対象 release: v1.2.0 completion gate

## 1. 目的
Shirube の `standard` / `strict` 品質モードを GitHub-backed workflow として固定し、複数エージェント、監査、PR、Issue、Check Run、Review の証跡を GitHub 上に集約する。

また、開発途中で設計判断や仕様変更が追加された場合に、producer が暗黙に実装を進めず、SPEC/IMPL/VERIFY/OPS と gate 状態へ明示的に取り込む手順を定義する。

## 2. 非目的
- GitHub なしの完全運用を `standard` / `strict` の同等機能として提供しない。
- `minimal` / offline draft を廃止しない。
- GitHub 以外の forge を v1.2.0 completion scope に含めない。
- 自然言語による「次にやるべきタスク」判断を SSOT にしない。

## 3. ユーザーストーリー
- As implementation lead, `shirube next --json` で GitHub Issues に基づく一意な次タスクを取得したい。
- As auditor, strict work の task ordering、PR review、gate result、design-change decision を GitHub 上の証跡から再検証したい。
- As operator, 途中で設計変更が入った時に、どの doc と gate を戻すべきかを CLI が示してほしい。
- As product owner, v1.2.0 完了時に「MCP として公開しても説明可能な品質境界」を持ちたい。

## 4. 機能要件

### 4.1 [SPEC-DOC4L-018-FR-001] GitHub-backed mode policy
`standard` / `strict` は GitHub-backed workflow を前提とする。

- task queue SSOT: GitHub Issues
- PR review / audit SSOT: GitHub PR reviews, PR comments, issue comments
- gate result SSOT: GitHub Check Runs
- local files: cache, hook support, offline draft, generated templates

`minimal` は GitHub なしで進行可能。ただし public MCP-quality release 判定には使えない。

### 4.2 [SPEC-DOC4L-018-FR-002] Adoption state model
Shirube は対象 repository を次の 3 状態に分類し、推奨コマンドと session boundary を明示する。

| State | 対象 | 判定条件 | Primary command | Start boundary |
|---|---|---|---|---|
| S1: greenfield | Shirube 未導入で、全く新規に開発する repository / directory | `.framework/` なし、既存 product code なし、または operator が新規作成を明示 | `shirube init <name>` → `shirube roles set ...` → `shirube start --feature <id>` | `start` が `.framework/current-session.json` を作成した時点 |
| S2: midstream adoption | Shirube 未導入だが、開発途中の既存 repository | `.git/` または既存 source/docs あり、`.framework/` なし | `shirube retrofit [path] --scan-only` → `shirube retrofit --generate` → `shirube roles set ...` → `shirube start --feature <id>` | `start` まで framework-led development ではない |
| S3: managed resume/update | Shirube 導入済み repository を再開、または最新版へ追従する | `.framework/` あり | `shirube update [path]` when needed → `shirube roles doctor` → `shirube start --resume` or `shirube start --force --feature <id>` | `resume` は既存 session を再有効化、`force` は明示的に session を切り替える |

`init`, `retrofit`, `update` は適用・更新であり、開発開始ではない。`standard` / `strict` の framework-led development は必ず `start` / `start --resume` / `start --force` のいずれかで session boundary を作る。

S2 では既存 code/docs を消さず、まず scan/report を出す。S3 では `.framework/config.json` の concrete role bindings と workflow settings を operator-owned として保存する。

### 4.3 [SPEC-DOC4L-018-FR-003] Strict start GitHub readiness
`shirube start --audit-level strict` は GitHub-backed readiness を検査する。

必須:
- git remote が GitHub repository を指す
- `gh` が利用可能で認証済み
- repo metadata を取得できる
- required role bindings が concrete かつ producer/authority 分離済み
- task queue / check-run / review evidence を記録できる構成

欠ける場合は `strict` を BLOCK する。`standard` は release/public work では BLOCK、移行中 dogfooding では warning を許容できるが、その場合は session に warning を記録する。

### 4.4 [SPEC-DOC4L-018-FR-004] Unique GitHub task sequence
GitHub Issues 上の task は一意で sortable な sequence value を持つ。

- canonical format: `WWWFFFFTTT`
- durable location: GitHub Issue metadata
- local `.framework/plan.json` / run-state は cache
- duplicate / missing sequence は autonomous run を BLOCK
- `shirube next --json` は exactly one next task を返す。曖昧なら exit non-zero。

### 4.5 [SPEC-DOC4L-018-FR-005] Resequence and insertion
task/PR/Issue 追加時は以下のどちらかを必須にする。

- gap allocation: 既存 sequence 間に空きがある場合のみ採番
- resequence: 空きがない場合、GitHub Issue metadata と local cache を同時に更新

`shirube resequence` は GitHub と local cache の両方を更新し、idempotent でなければならない。

### 4.6 [SPEC-DOC4L-018-FR-006] Design-change intake
開発途中で設計変更、製品判断、品質境界変更、外部指示追加が入った場合、Shirube は design-change intake として扱う。

変更分類:
- SPEC impact: product behavior, policy, public contract, role model, security/trust boundary
- IMPL impact: module layout, algorithm, storage, integration, CLI behavior
- VERIFY impact: acceptance criteria, regression, E2E, audit checks
- OPS impact: rollout, migration, operator workflow, incident handling

SPEC impact がある場合、producer は実装を続行できない。該当 docs 更新と `/gate-design` 通過まで active implementation を BLOCK する。

### 4.7 [SPEC-DOC4L-018-FR-007] Active session drift recording
active session 中の design change は `.framework/current-session.json` または GitHub issue/PR comment に以下を記録する。

- change id
- source: user / reviewer / auditor / external instruction file / issue / PR
- impact classification
- affected docs
- required gate rollback
- authority decision
- resumedAt

raw secret や private instruction body は保存しない。

### 4.8 [SPEC-DOC4L-018-FR-008] Update safety
`shirube update` は operator-owned config を上書きしない。

- concrete `roles.bindings` は保持
- missing required role は placeholder で補完
- placeholder は concrete binding を置換しない
- `.framework/config.json` の operator edits は preserve-first

## 5. インターフェース

### 5.1 CLI policy
```text
shirube start --audit-level strict
  GitHub readiness failed -> exit 1
  role readiness failed -> exit 1
  sequence readiness failed -> exit 1

shirube next --json
  one eligible task -> JSON
  duplicate/missing/ambiguous seq -> exit 1

shirube resequence
  updates GitHub Issues + local cache

shirube design-change record
  records active-session design drift and required gate rollback
```

`design-change` command name is provisional; implementation may fold it into `status`, `start`, or `gate` if the same behavior is mechanically enforced.

### 5.2 GitHub task metadata
The implementation must choose one durable GitHub representation and document it:

- issue body `adf-meta` JSON
- label such as `seq:1000100010`
- GitHub Project field

The selected representation is the SSOT. Other representations are cache or display.

## 6. 非機能要件
- No silent fallback from GitHub SSOT to local SSOT in `strict`.
- All checks must be deterministic and script-controlled.
- GitHub API failure in `strict` is fail-closed.
- Local/offline mode must clearly mark itself as non-public-quality.

## 7. 受入基準
```gherkin
Feature: GitHub-backed strict workflow
  Scenario: strict start without GitHub readiness
    Given a project without authenticated GitHub access
    When the operator runs "shirube start --audit-level strict"
    Then the command exits non-zero
    And the session is not marked ready

  Scenario: duplicate task sequence blocks autonomous next
    Given two open GitHub task issues with the same seq
    When the agent runs "shirube next --json"
    Then the command exits non-zero
    And the output explains the duplicate seq

  Scenario: design change during implementation
    Given a strict session is in implementation phase
    When a SPEC-impacting product decision is recorded
    Then active implementation is blocked
    And required docs and gate rollback are listed
```

## 9. リスクと緩和
- GitHub dependency increases setup friction: keep `minimal` offline mode explicit.
- Sequence metadata drift: choose one GitHub SSOT and validate all mirrors.
- Design-change process may feel heavy: allow small-change mode only when no SPEC/contract/security/role impact exists.

## 10. 制御機構選定原則
script 選定根拠: Default mechanism is script-controlled CLI and GitHub Checks because GitHub readiness, queue validation, resequence, and design-change blocking must be deterministic and re-runnable without LLM judgment.

Hook fallback is allowed only for pre-tool or session-bound blocking where CLI cannot intercept. This spec does not adopt a new Hook as the primary control mechanism; any future Hook adoption must explicitly satisfy the unavoidable 4 case rule.

| FR | 機構 | 根拠 |
|---|---|---|
| FR-001 | CLI + docs | mode policy |
| FR-002 | CLI + docs + state detection | adoption state must choose the correct lifecycle command |
| FR-003 | CLI + GitHub API | strict readiness must fail closed |
| FR-004 | CLI + GitHub API | task queue SSOT |
| FR-005 | CLI + GitHub API | deterministic renumbering |
| FR-006 | CLI/gate | active design drift control |
| FR-007 | session file + GitHub comment | audit trail |
| FR-008 | CLI update engine | config preservation |
