## 4. 機能要件 (Core) [必須]

### 4.1 [SPEC-DOC4L-012] Test Coverage Gap Discipline

[文献確認: proposal §SPEC-DOC4L-012]

#### FR-012-1: 4 ギャップ評価
新規 feature 起票時に SPEC §「Test Coverage Gap」 section で以下 4 種を必須評価:

1. **Coverage gap**: 仕様カバレッジ (要件 → test 漏れ)
2. **Environment gap**: 環境カバレッジ (dev / staging / production の差異漏れ)
3. **Tooling gap**: 検証ツール不足 (mock では verify 不可な real env 検査漏れ)
4. **Skill gap**: 知識不足 (担当 dev-bot の expertise 不足部分)

#### FR-012-2: gap 解消アクション
各 gap に対して以下を Issue / PR で明示:
- gap 内容 + 影響範囲
- 解消アクション (test 追加 / 環境用意 / tool 導入 / pair プログラミング 等)
- 解消 verify 方法

### 4.2 [SPEC-DOC4L-013] Behavior-Driven Specification

[文献確認: proposal §SPEC-DOC4L-013]

#### FR-013-1: AC ↔ test 1:1 紐付け
SPEC §7「受入基準」の各 Gherkin Scenario に対応する test を `tests/acceptance/` に **1:1** で配置:
- Scenario name = test ID (例: `AC-AGENTCOM-001`)
- 各 test は AC の Given/When/Then を 1:1 で実装
- AC 増減 → test も同期 (PR review で検証)

#### FR-013-2: AC carrier file
SPEC.md 内の Gherkin block を script で grep 抽出可能な format で固定 (規約: `### AC-<FEATURE>-<NNN>` heading + 直下に gherkin code block)。

### 4.3 [SPEC-DOC4L-014] Property-Based Invariant Testing

[文献確認: proposal §SPEC-DOC4L-014]

#### FR-014-1: invariant 定義
重要 feature について SPEC §「Invariants」 section で invariant を明示:
- ID: `INV-<FEATURE>-<NNN>`
- 内容: 入力空間に対して常に成立すべき property
- 反例検出時の挙動

#### FR-014-2: Property-Based test
各 invariant に対して `tests/property/` に **fast-check** (or 同等) で 1000+ 通り入力 verify:
- arbitrary input generator
- assertion で invariant verify
- 反例 (counterexample) 検出時 stack trace + input を記録

### 4.4 [SPEC-DOC4L-015] Traceability Matrix

[文献確認: proposal §SPEC-DOC4L-015]

#### FR-015-1: 要件 ↔ test ↔ code link
全 feature で以下 3 者 link を `traceability.csv` (or 同等) で機械可読:
- 要件 ID (FR-XXX or AC-XXX)
- test ID (test file path + test name)
- code 範囲 (file path + line range or symbol)

#### FR-015-2: drift 検出 CLI
`framework trace verify` (v1.2.0 既実装、PR #103) を本 spec で **3 者 link 完全性 verify** に拡張:
- 要件があって test なし → fail
- test があって code 無関係 → warn
- code 変更で trace 切れ → fail

#### FR-015-3: Traceability Matrix 起票
各 SPEC merge 時に traceability matrix の差分を PR description に明示 (auto-generated 推奨)。

## 7. 受入基準 (Acceptance Criteria) [必須・Gherkin形式]

### 7.1 [SPEC-DOC4L-012] Coverage Gap 評価

```gherkin
Feature: 4 ギャップ評価
  Scenario: 新 feature SPEC 起票時
    Given SPEC.md に §「Test Coverage Gap」 section 不在
    When framework gate validate spec を実行
    Then exit 2、stderr に "Coverage Gap section missing"
```

### 7.2 [SPEC-DOC4L-013] AC ↔ test 1:1

```gherkin
Feature: AC ↔ test 1:1
  Scenario: AC 増えた時
    Given SPEC.md に AC-FEATURE-005 が新規追加
    When framework trace verify を実行
    Then exit 2 (test 不在)
    And stderr に "AC-FEATURE-005 has no corresponding test"
```

### 7.3 [SPEC-DOC4L-014] Property-Based invariant

```gherkin
Feature: Invariant 1000 通り verify
  Scenario: invariant 違反検出
    Given INV-FEATURE-001 が定義
    And 1000 通り input で verify
    When 反例検出
    Then test fail + counterexample 記録
```

### 7.4 [SPEC-DOC4L-015] Traceability drift

```gherkin
Feature: 3 者 link 完全性
  Scenario: code 削除で trace 切れ
    Given traceability.csv に FR-001 ↔ test_X ↔ src/foo.ts:42 が記録
    And src/foo.ts:42 が削除
    When framework trace verify
    Then exit 2、stderr に "FR-001 trace broken: src/foo.ts:42 not found"
```
