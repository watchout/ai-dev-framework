### 4.2 [SPEC-DOC4L-011] Verify 4-Evidence Discipline

[文献確認: proposal §SPEC-DOC4L-011]

#### FR-011-1: 4-Evidence ルール
CTO 等 verify 担当の「完了」claim には以下 4 種類の証跡が必須:

1. **process evidence**: launchd / process 起動状態 (`launchctl list` / `pgrep`)
2. **DB evidence**: `message_queue` / `agents` table の状態 (psql query 出力)
3. **log evidence**: stdout/stderr log の最新 N 行 (実 file から)
4. **tmux pane evidence**: tmux capture-pane で実 bot 応答

各証跡は **CLI コマンド + 出力** をそのまま記録 (pasted screenshot 不可、再現可能性必須)。

#### FR-011-2: routine 化
CTO L3 sanity / Phase 0 verify で 4-Evidence を必須化、欠如時は merge gate failure。

## 5. インターフェース (Contract) [必須]

### 5.1 launchd integration test 契約 [SPEC-DOC4L-010]

```bash
# tests/e2e/launchd-integration.test.sh
launchctl load com.agent-comms.state-daemon.plist
sleep 5
launchctl list | grep com.agent-comms.state-daemon  # → 状態 0 (running)
kill -9 $(pgrep -f state-daemon)
sleep $((ThrottleInterval + 5))
launchctl list | grep com.agent-comms.state-daemon  # → restart 確認
launchctl unload com.agent-comms.state-daemon.plist
```

### 5.2 E2E wake test 契約 [SPEC-DOC4L-010]

```bash
# tests/e2e/wake-verification.test.sh
psql -c "INSERT INTO message_queue (agent_id, status, ...) VALUES ('test-bot', 'pending', ...)"
# state-daemon が tmux send-keys 発火する想定 (sleep + verify)
sleep 2
tmux capture-pane -t test-bot -p | grep "check inbox"  # → 受領確認
psql -c "SELECT last_wake_attempt_at FROM message_queue WHERE id=$ID"  # → not null
```

### 5.3 4-Evidence 証跡 schema [SPEC-DOC4L-011]

CTO L3 sanity の verify report は以下 4 section を含むこと:

```markdown
## Verify 4-Evidence (commit <sha>)
### 1. Process evidence
$ launchctl list | grep state-daemon
... (output)
### 2. DB evidence
$ psql -c "SELECT ..."
... (output)
### 3. Log evidence
$ tail -50 /var/log/state-daemon.out.log
... (output)
### 4. Tmux pane evidence
$ tmux capture-pane -t <session> -p
... (output)
```

## 6. 非機能要件 (Detail) [必須]

### 6.1 性能
- E2E test 1 回実行時間 < 60 秒 (CI で許容範囲)
- 4-Evidence 収集時間 < 30 秒 (CTO routine への組込負荷)

### 6.2 可用性
- macOS dev fleet 前提 (Linux CI では launchd test skip)
- 4-Evidence は手動収集が default、自動化は別 PR

### 6.3 セキュリティ要件 [app/api プロファイルで必須]

#### 6.3.1 脅威モデル (STRIDE)
| カテゴリ | 該当内容 |
|---|---|
| Spoofing | N/A (内部 dev 環境) |
| Tampering | 4-Evidence 証跡の改竄 → git 管理 + PR 経由のみ |
| Repudiation | 全 verify を PR comment + commit message に記録 |
| Information Disclosure | log evidence に PII / secret を含めないこと |
| DoS | E2E test の頻度制限 (CI で 1 回 / push) |
| Privilege Escalation | N/A |

#### 6.3.2 OWASP: A09 Logging Failures (4-Evidence で対応)、A04 Insecure Design (本 spec で structural 解消)

#### 6.3.3 データ分類
- 4-Evidence 出力: [x] 公開 (repo PR comment)
- log evidence: [ ] 機密 (PII filter 必須、6.3.1 per)

### 6.4 監査ログ要件
