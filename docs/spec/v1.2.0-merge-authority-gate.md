# SPEC: Shirube Merge Authority Gate

## 0. Meta
- Created: 2026-05-21
- Origin Issue: #184
- Related Issue: #183
- Target release: v1.2.0 public MCP-quality completion gate

## 1. Purpose
Prevent governed Shirube pull requests from being merged unless configured authority roles have produced deterministic, current-head approval evidence.

#183 showed that documented L3 merge authority is insufficient when merge paths are not mechanically enforced. This spec defines a required merge-authority check that applies to web UI, `gh pr merge`, connector-based merge, and `shirube merge`.

## 2. Non-Goals
- Do not use LLM judgment or natural-language interpretation as merge authority.
- Do not treat labels as approval evidence.
- Do not treat check runs as human authority evidence.
- Do not replace GitHub branch protection.
- Do not allow `auto_publish` to bypass L3/release authority.

## 3. Terms
| Term | Meaning |
|------|---------|
| Governed PR | Any PR targeting a protected branch in a Shirube-managed repository. |
| Authority evidence | A deterministic GitHub approval signal from a configured role actor, tied to the current PR head SHA. |
| Gate result | The `shirube merge-authority` check run result. |
| Route label | A label that scales required evidence, such as `route:fast-merge` or `route:ceo-approval`. |

## 4. Functional Requirements

### 4.1 [SPEC-DOC4L-019-FR-001] Required status check
Protected branches in governed repositories must require a GitHub status check named `shirube merge-authority`.

This is the only enforcement point that reliably covers:
- GitHub web UI merge.
- `gh pr merge`.
- connector/API merge.
- `shirube merge`.

Documentation-only merge rules are insufficient.

### 4.2 [SPEC-DOC4L-019-FR-002] Canonical authority evidence
Canonical authority evidence is the effective GitHub PR review state for a configured authority actor. The actor's latest non-dismissed review on the current `headRefOid` must be `APPROVED`.

Rules:
- Approval evidence must be tied to the current `headRefOid`.
- If a bound actor's latest current-head review is `CHANGES_REQUESTED`, `COMMENTED`, dismissed, or absent, that actor has no valid approval evidence.
- If a bound actor approved an older head and has no current-head approval, that approval is stale and invalid.
- Labels may route requirements but must not count as approval evidence.
- Check runs are gate result SSOT, not human authority evidence.
- Comments are not approval evidence for v1.2.0 MVP.

Future machine-readable comments may be considered only if they include role, actor, and head SHA, and are authored by a configured authority actor.

### 4.3 [SPEC-DOC4L-019-FR-003] Role identity binding
The merge-authority evaluator reads `.framework/config.json`.

Required authority roles:
- `l3_governance_owner`
- `release_owner`
- `human_approver`

For merge authority, those roles must resolve to GitHub identities:
- `github_user:<login>`
- `github_team:<org/team-slug>`

Existing role target types such as `external`, `local_agent`, or `mcp_agent` do not satisfy merge authority unless they are explicitly mapped to a GitHub identity in config.

Producer roles (`implementation_lead`, `worker_pool`) can never satisfy authority evidence. If role separation validation fails, merge authority fails closed.

### 4.4 [SPEC-DOC4L-019-FR-004] Stale evidence
Authority evidence is valid only for the current PR head SHA.

The evaluator must fail closed when:
- the approval belongs to an older head SHA;
- the current head has no approval from the configured role actor;
- GitHub review data is unavailable or ambiguous.

Repositories must enable GitHub branch protection option "dismiss stale pull request approvals when new commits are pushed" where available. The check still validates current-head evidence directly.

### 4.5 [SPEC-DOC4L-019-FR-005] Required authority matrix
The check runs on every governed PR. The final required roles are the union of route requirements, workflow policy requirements, and audit-level requirements.

Route defaults:
- `route:fast-merge`: require `l3_governance_owner`.
- `route:ceo-approval`: require `l3_governance_owner`, `release_owner`, and `human_approver`.
- no route or ambiguous route: fail closed as `route:ceo-approval`.

Workflow policy interaction:
- `draft_only`: remote merge is blocked.
- `approval_required`: require valid `human_approver` evidence in addition to route requirements.
- `auto_publish`: may skip `human_approver` only when route does not require it, but must never skip `l3_governance_owner` or `release_owner` when required.

Audit-level interaction:
- `strict`: require `l3_governance_owner`.
- `standard`: require `release_owner`; require `l3_governance_owner` when route/path is governed.
- unknown or ambiguous audit level: fail closed.

The implementation must support all three authority roles for v1.2.0/#184 when required by the union matrix. `release_owner` and `human_approver` validation are not optional follow-up for routes or policies that require them.

### 4.6 [SPEC-DOC4L-019-FR-006] Fail-closed behavior
The merge-authority check must fail closed for:
- missing role binding;
- placeholder role binding;
- non-GitHub authority role binding;
- wrong actor approval;
- stale approval;
- label-only approval;
- producer-as-approver;
- missing route or ambiguous route;
- GitHub API failure;
- malformed `.framework/config.json`.

### 4.7 [SPEC-DOC4L-019-FR-007] CLI behavior
`shirube merge doctor` is read-only. It evaluates the same core logic as CI and reports PASS/BLOCK plus missing evidence.

`shirube merge` must always run doctor immediately before merge. It must not merge when doctor blocks.

The CLI is convenience, not the root enforcement point. Branch protection and the required check remain mandatory.

## 5. MVP Slice
The v1.2.0 MVP must implement:
- `shirube merge-authority` check logic for current-head GitHub review evidence.
- GitHub identity resolution for `l3_governance_owner`, `release_owner`, and `human_approver`.
- final required-role union across route, workflow policy, and audit level.
- effective latest non-dismissed current-head review reduction per bound actor.
- fail-closed behavior for missing/wrong/stale/placeholder evidence.
- branch protection setup docs.
- tests for missing, wrong actor, stale, dismissed, changes-requested, producer-as-approver, label-only, and valid approval.

`shirube merge` CLI convenience may land after the MVP if the required status check path already blocks the #183 violation class. Team-membership expansion may also follow if MVP starts with explicit `github_user` bindings.

## 6. Completion Criteria
- A PR without required L3/release/human approval evidence is blocked.
- A PR with valid current-head approval from configured authority actors passes.
- Evidence is tied to the current PR head SHA.
- Labels alone never pass the gate.
- Producer actors cannot satisfy authority roles.
- The check is deterministic and does not rely on LLM interpretation.
- Branch protection setup includes `shirube merge-authority` as a required check.
- Direct `gh pr merge` and connector merge paths are blocked by branch protection unless the required check has passed.

## 7. Acceptance Criteria

### 7.1 Missing authority evidence blocks
```gherkin
Given a governed pull request targeting a protected branch
And the pull request has no current-head APPROVED review from the configured l3_governance_owner
When shirube merge-authority evaluates the pull request
Then the check fails closed with a missing_l3_governance_owner reason
```

### 7.2 Valid current-head authority evidence passes
```gherkin
Given a governed pull request targeting a protected branch
And l3_governance_owner resolves to github_user:cto-login
And cto-login's latest non-dismissed review on the current headRefOid is APPROVED
When shirube merge-authority evaluates the pull request
Then the check passes
```

### 7.3 Stale, dismissed, or wrong actor evidence blocks
```gherkin
Given a governed pull request with an APPROVED review from an unbound actor, an older headRefOid, or a dismissed review
When shirube merge-authority evaluates the pull request
Then the check fails closed and reports invalid authority evidence
```

### 7.4 Label-only approval blocks
```gherkin
Given a governed pull request with route labels and approval-looking labels
And no current-head APPROVED review from the configured authority actor
When shirube merge-authority evaluates the pull request
Then the check fails closed because labels are not approval evidence
```

## 8. Interfaces

### 8.1 CLI
```bash
shirube merge doctor <pr-number>
shirube merge <pr-number>
shirube merge-authority --pr <pr-number> --json
```

Exit codes:
- `0`: authority evidence valid for current head.
- `1`: deterministic BLOCK.
- `2`: configuration or GitHub API failure; fail closed.

### 8.2 GitHub Actions
The required check name is `shirube merge-authority`.

The check must run on `pull_request` events that can alter PR route, head, or draft state:
- opened;
- synchronize;
- reopened;
- ready_for_review;
- labeled;
- unlabeled.

The check must also run on `pull_request_review` events that can alter review authority state:
- submitted;
- edited;
- dismissed.

### 8.3 Configuration
The evaluator reads `.framework/config.json` as the role and workflow SSOT.

## 9. Rollout

### 9.1 Shirube repository
1. Add the check workflow.
2. Configure branch protection to require `shirube merge-authority`.
3. Enable stale approval dismissal.
4. Verify #183 violation class cannot be reproduced.

### 9.2 Downstream repositories
Totonoe, kodama, and other downstream repositories may continue design/prep work before the gate is installed.

They must not claim strict public MCP-quality merge governance until the required status check and branch protection are active.

### 9.3 Risks and Mitigations

| Risk | Mitigation |
|------|------------|
| GitHub team membership API unavailable | fail closed and require explicit `github_user` binding for MVP |
| branch protection not configured | repository cannot claim public MCP-quality strict governance |
| approval from wrong actor | compare GitHub review author with configured authority identity |
| approval becomes stale after head change | require current `headRefOid` evidence and enable stale dismissal |
| direct merge bypass | branch protection required check blocks web, CLI, and connector merge paths |
| auto-merge workflow uses labels only | remove or harden the workflow to require `shirube merge-authority` |

## 10. 制御機構選定原則

script 選定根拠: merge authority は deterministic script / GitHub Actions / GitHub API evaluation で判定する。approval text を LLM が解釈する方式は採用しない。

Hook 選定理由: MVP は Hook 不採用。merge path の共通 enforcement point は GitHub branch protection required check であり、Hook は web UI / `gh` / connector merge を横断して強制できない。

| Control Surface | Mechanism | Rationale |
|-----------------|-----------|-----------|
| Merge authority decision | deterministic GitHub API evaluation | Natural-language approval parsing is not reliable enough for public MCP-quality governance. |
| Evidence source | current-head GitHub PR reviews | Reviews provide actor identity and commit association. |
| Route selection | labels as routing inputs only | Labels are mutable routing metadata, not authority evidence. |
| Gate result | GitHub required status check | Branch protection is the only enforcement point shared by web UI, `gh`, connectors, and `shirube merge`. |
| CLI preview | `shirube merge doctor` read-only | Operators need local/debug visibility without changing PR state. |
| CLI merge | `shirube merge` runs doctor internally | Convenience path must use the same deterministic evaluator as CI. |

Hook usage:
- No runtime Hook is required for the merge-authority MVP.
- The enforcement mechanism is GitHub branch protection plus a deterministic status check.
- If a future Hook is proposed, it must fit an allowed runtime-control case and cannot replace the required status check.

LLM role:
- LLMs may summarize missing evidence for humans.
- LLMs must not decide whether authority evidence is valid.
- LLMs must not merge or bypass branch protection.
