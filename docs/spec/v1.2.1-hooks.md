# SPEC: ADF v1.2.1 — Claude Code Hooks 基盤統合

## 0. メタ
- 作成日: 2026-05-08
- 関連 ADR: ADF v1.2.0 (4 層体系 + 原則 0)
- 起点 proposal: `proposals/v1.2.x/adf-v1.2.x-spec-proposal.md` SPEC-DOC4L-008
- 引き渡し: ARC (CTO directive `579bf30b` per、CEO `4c349483`)
- Honesty: [検証済] / [文献確認] / [推測] ラベル全 claim 必須

## 1. 目的 (Goals) [必須]

ADF v1.2.0 の Gate / Validator (document 検証) に加え、bot session 内の **runtime 制御** として Claude Code 公式 hooks を ADF 配下に統合する。

CTO 自認の「memory rule に書いたが守らない」failure mode、および 4 bot 全員が自認した「動くと思うが確認していない」発言 multi-occurrence を、**bot の意志に依存しない technical enforcement** で構造的に防止する。

## 2. 非目的 (Non-goals) [必須]

- v1.2.2 範囲の高度 hook (label check / citation 検証 / post-merge full verify / verdict persist) は本 spec out of scope (SPEC-DOC4L-009 で定義)
- Plan Mode 必須化は v1.2.2 に分離
- Claude Code hooks 仕様自体の拡張は out of scope (公式仕様遵守のみ)
- bot LLM の prompt rule (advisory CLAUDE.md) は本 spec で扱わない (script 制御原則 0 と直交)

## 3. ユーザーストーリー [必須]

- **As an ADF user (project owner)**、`framework hook init` 1 コマンドで public template が `.claude/settings.json` に展開されるので、各 bot が同一 enforcement で稼働する。
- **As a CTO/lead-bot**、PR review 時に「post-merge skip」「main 直 push」「admin merge」「lint 漏れ」が hook block で技術的に発生不能になり、leadership 圧で防いでいた違反が 0 件化する。
- **As an ADF maintainer**、hook 設定 / script は ADF 配下で集約管理され、`framework hook validate` で syntax / 公式仕様適合を機械検証できる。

## 4. 機能要件 (Core) [必須]

### 4.1 [SPEC-DOC4L-008-FR-001] settings.json template 提供 (F1)

ADF が `templates/project/.claude/settings.json` を提供。`framework init` 実行時に対象 repo の `.claude/settings.json` に merge 配置。

template 内容は §4.4 で定義する 4 種 hook 構成を含む。既存 `.claude/settings.json` がある場合は **既存 hook を保持しつつ ADF 提供 hook を追加 merge** (key conflict は ADF 側 default、override mode は config flag)。

### 4.2 [SPEC-DOC4L-008-FR-002] framework hook サブコマンド (F2)

```
framework hook init       # template から settings.json 生成 (既存と merge)
framework hook validate   # 既存 hook 設定の syntax check + Claude Code 公式仕様適合
framework hook test       # hook の動作確認 (dry-run、各 hook を順次トリガー)
framework hook list       # 現在 active な hook を一覧表示
```

`framework hook validate` failure は exit code != 0、CI で gate として使用可能。

### 4.3 [SPEC-DOC4L-008-FR-003] Gate と Hook の役割分担明示 (F3)

ADF SPEC.md / IMPL.md template に「検証層の構造」章を追加:

- **層 A: ADF Gate** (document 検証、Phase 境界、CLI 明示呼出、v1.2.0 で実装済)
- **層 B: Claude Code Hooks** (runtime 検証、bot session 内 auto intercept、bot 意志に依存しない、v1.2.1 で導入)

両層は独立だが ADF が両方の設定 / 監査ログを集約管理する。詳細は IMPL-DOC4L-008 §2 を参照。

### 4.4 [SPEC-DOC4L-008-FR-004] 基本 hook 4 種 template (F4)

#### F4-1: PostToolUse (編集後の品質検証)

[文献確認 https://code.claude.com/docs/en/hooks] file 編集後 (`Write` / `Edit` / `MultiEdit`) に prettier / eslint / type-check を自動実行、失敗時 exit 2 で次の処理を block。

- 想定: `prettier` / `eslint` / `tsc` (`typescript` package) が `$PATH` で利用可能であること。未 install の場合 `command -v` で graceful skip + warning log (OPS §5.3 runbook 参照)。

#### F4-2: Stop (完了前の test 強制実行)

[文献確認 Boris Tip 2] bot が完了宣言する直前に test suite を実行、fail なら `{"decision":"block","reason":"Test suite must pass before completion"}` で完了を block。`stop_hook_active=true` で無限 loop 防止。

#### F4-3: SessionStart (spec / state 自動注入)

`startup` / `resume` / `compact` で発火。直近 modified spec / 関連 ID (SPEC-* / IMPL-*) を `additionalContext` JSON で stdout 出力、bot context に注入。

#### F4-4: PreToolUse (危険コマンド block)

`Bash` matcher で発火。`gh pr merge --admin` / `git push origin main` / `rm -rf` / `DROP TABLE` / `chmod 777` 等のパターンを検出、exit 2 で tool 実行 block + stderr に reason 出力。

### 4.5 [SPEC-DOC4L-008-FR-005] hook script の portable path

全 hook script は `$CLAUDE_PROJECT_DIR` prefix で portable path 化。bot fleet 横断で同一 settings.json template が動作可能。

## 5. インターフェース (Contract) [必須]

### 5.1 CLI 契約

```
framework hook init [--force]
  exit 0: 成功
  exit 2: 既存 settings.json と衝突、--force 必須
  exit 1: その他 error

framework hook validate [--strict]
  exit 0: 公式仕様適合
  exit 2: syntax / 仕様違反検出
  exit 1: その他 error

framework hook test [--hook=<name>]
  exit 0: 全 hook が trigger / block / pass の expected behavior 通り
  exit 2: 1 つ以上 expected と乖離

framework hook list
  exit 0: stdout に table 形式 (event / matcher / command / status)
```

### 5.2 hook script 契約 (Claude Code 公式)

- `exit 0` = pass、後続処理続行
- `exit 2` = block、stderr の reason を bot に表示
- Stop hook の JSON output: `{"decision":"block","reason":"..."}`
- SessionStart の JSON output: `{"additionalContext":"..."}`

詳細は IMPL-DOC4L-008 §2 / §4 を参照。

### 5.3 イベント / メッセージ [該当時]

該当なし。本 spec は document + script 配置層、外部メッセージング系統は変更しない。

## 6. 非機能要件 (Detail) [必須]

### 6.1 性能

- hook 実行時間 < 5 秒 (per event)、超過時は warn log
- `framework hook validate` < 3 秒 (single repo、template + 既存 settings 比較)

### 6.2 可用性 (SLO)

- hook script 失敗時、ADF Gate (層 A) は影響を受けない (層 B 独立)
- hook script 自体の crash は bot session を block しないこと (script-level fail-safe)

### 6.3 セキュリティ要件 [app/api プロファイルで必須]

#### 6.3.1 脅威モデル (STRIDE)

| カテゴリ | 該当内容 |
|---|---|
| Spoofing | bot identity の偽装は本 spec scope 外 (Claude Code session が SoT) |
| Tampering | hook script を悪意改変 → 検知は branch protection + PR review (層 A 範囲) |
| Repudiation | hook 実行 log を audit trail として保持 (`.framework/hook-log/`) |
| Information Disclosure | hook script は project 内のみ読込、外部 API 呼出は禁止 (template default) |
| Denial of Service | hook 実行時間 > 5s で warn、無限 loop は `stop_hook_active` で防止 |
| Elevation of Privilege | `gh pr merge --admin` block (F4-4) で governance bypass を技術的に防ぐ |

#### 6.3.2 OWASP Top 10:2021 マッピング

- A04 Insecure Design: Hook 4 case 限定、それ以外は script 制御 (Notion 共有原則)
- A08 Integrity Failures: hook script は repo にコミット、`framework hook validate` で改竄検知
- A09 Logging Failures: 全 hook 実行を `.framework/hook-log/` に記録
- 他項目: N/A (本 spec は internal control mechanism、認証認可 / 入力検証は範囲外)

#### 6.3.3 データ分類

- 本 feature が扱うデータ:
  - [ ] PII
  - [ ] PCI
  - [ ] 機密 (script 内に secret を埋め込まないこと、Forbidden)
  - [x] 公開 (hook script source は repo 公開)

### 6.4 監査ログ要件 [該当時]

全 hook 実行を `.framework/hook-log/{date}.jsonl` に追記 (event / matcher / exit_code / duration_ms / reason)。`framework hook log` で集計可能 (本 spec FR ではない、別 task)。

## 7. 受入基準 (Acceptance Criteria) [必須・Gherkin形式]

### 7.1 [SPEC-DOC4L-008-FR-001] settings.json template 提供

```gherkin
Feature: settings.json template 提供
  Scenario: クリーン repo に init
    Given .claude/settings.json が存在しない
    When framework hook init を実行
    Then .claude/settings.json が生成され、PostToolUse / Stop / SessionStart / PreToolUse の 4 種を含む
    And exit code は 0

  Scenario: 既存 settings.json と merge
    Given .claude/settings.json に既存 PreToolUse hook 1 件
    When framework hook init を実行
    Then 既存 PreToolUse hook は保持される
    And ADF 提供の 4 種 hook が追加される
    And exit code は 0
```

### 7.2 [SPEC-DOC4L-008-FR-002] framework hook validate

```gherkin
Feature: hook validate
  Scenario: syntax 違反検知
    Given settings.json に matcher 欠落の hook
    When framework hook validate を実行
    Then exit code は 2
    And stderr に "matcher missing" が出力される
```

### 7.3 [SPEC-DOC4L-008-FR-004-2] Stop hook が test fail を block

```gherkin
Feature: Stop hook
  Scenario: test fail 時に完了 block
    Given test suite が fail する状態
    When bot が completion を試行
    Then Stop hook が exit 2 + reason JSON を返す
    And bot session に "Test suite must pass before completion" が表示される
```

### 7.4 [SPEC-DOC4L-008-FR-004-4] PreToolUse が `gh pr merge --admin` を block

```gherkin
Feature: PreToolUse 危険コマンド block
  Scenario: admin merge block
    Given bot が "gh pr merge 123 --admin" を Bash で実行しようとする
    When PreToolUse hook が発火
    Then exit code は 2
    And stderr に "governance violation: admin merge" が出力される
    And tool は実行されない
```

### 7.5 [SPEC-DOC4L-008-FR-005] portable path

```gherkin
Feature: portable path
  Scenario: 別 repo でも動作
    Given $CLAUDE_PROJECT_DIR 経由で hook script が呼ばれる
    When 別の repo で同一 settings.json template を install
    Then hook 全種が同一の動作をする
```
