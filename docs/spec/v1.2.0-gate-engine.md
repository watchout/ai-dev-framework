# SPEC: Shirube Gate Engine

## 0. Meta
- Created: 2026-05-23
- Origin Issue: #197
- Implementation issue: #198
- Target release: v1.2.x public-grade Gate Engine baseline

## 1. Purpose
Define the public-grade Gate Engine architecture that drives an LLM-led project from an explicit product goal to verified delivery. The Gate Engine unifies goal contracts, phase planning, feature/task decomposition, SPEC/IMPL/VERIFY/OPS readiness, reproducible Context Packs, implementation and audit gates, AI Change Records, GitHub Check projection, merge authority, post-merge verification, MCP surfaces, GitHub-backed workflow, and local-only operation.

Shirube is not only a gate collection. Its core value is correcting LLM local optimization by keeping every phase, feature, task, PR, audit, and post-merge result traceable to a declared V0/V1 goal contract.

This document is a specification baseline only. It introduces no runtime behavior change and must not make existing `shirube discover -> shirube generate` stricter.

## 2. Non-Goals
- Do not implement new blocking enforcement in this PR.
- Do not require GitHub for local-only projects.
- Do not create a second role or authority model.
- Do not replace `workflow-config.ts` or `merge-authority.ts`.
- Do not make hooks the canonical source of workflow state.
- Do not allow LLM output to decide phase completion, gate pass/fail, or exception approval.
- Do not let a feature catalog, task list, SPEC, or PR stand in for a missing goal contract.
- Do not apply this as a single oversized implementation PR.

## 3. Core Invariants
1. A V0/V1 Goal Contract is the anchor for planning and completion claims.
2. Shirube owns workflow state mechanically.
3. CLI, MCP, GitHub automation, and local-only usage consume the same state/rule model.
4. Phase transitions require deterministic evidence.
5. A GitHub Issue is intake evidence, not hearing completion and not goal approval.
6. Feature catalogs and task DAGs must trace to goal sufficient conditions.
7. Each AI task must run from a reproducible Context Pack, not a generic prompt.
8. SPEC/IMPL/VERIFY/OPS, implementation, audits, PRs, and post-merge verification must preserve goal traceability.
9. Every AI-generated or AI-assisted PR must be explainable through an AI Change Record.
10. Role readiness comes from `workflow-config.ts`.
11. Release and merge authority come from `merge-authority.ts`.
12. GitHub Checks are the primary enterprise adoption surface for remote workflows.
13. Provider/tool permissions are policy evidence, not hidden runtime assumptions.
14. Every gate decision is deterministic, explainable, and evidence-backed.
15. Enforcement rolls out as observe -> warn -> block by profile.

## 4. Enterprise Adoption Target
#211 sets the backcast target for this design:

```text
Shirube = AI PR Safety Gate + Evidence Control Plane
```

The adoption-worthy promise is not that Shirube writes code. Coding agents already do that. Shirube must prove that AI-generated changes are safe, traceable, reviewable, reproducible, and mergeable inside real software organizations.

Required enterprise-grade capabilities:

- agent-neutral governance across Claude Code, Codex, Cursor, Copilot, Gemini, internal agents, and generic CLI agents;
- reproducible Context Pack for every AI task;
- AI Change Record for every AI-assisted PR;
- GitHub Check / PR comment projection for trace, evidence, authority, and risk status;
- provider/tool policy evidence, including permission mode, allowed tools, denied tools, network/sandbox posture, and MCP server/tool configuration;
- measurable operational metrics such as trace coverage, complete AI Change Record rate, blocked spec-unlinked changes, missing-test-evidence blocks, risky tool calls, and time-to-merge impact.

## 5. Goal-Driven Lifecycle
Canonical lifecycle:

```text
adoption
  -> goal_contract
  -> phase_plan
  -> feature_catalog
  -> task_dag
  -> feature_spec
  -> impl_design
  -> context_pack
  -> provider_tool_policy
  -> pre_impl_audit
  -> implementation
  -> ai_change_record
  -> post_impl_audit
  -> github_check_projection
  -> merge_authority
  -> post_merge_verify
  -> goal_progress_review
```

### Goal Contract
A Goal Contract is required before a project can claim V0/V1 direction or strict/public-grade readiness. It must define:

- target version: `V0`, `V1`, or another explicit release goal;
- problem and intended outcome;
- sufficient conditions for goal completion;
- non-goals and out-of-scope boundaries;
- measurable success indicators where applicable;
- required artifacts and verification evidence;
- owner/approver role;
- trace links to phase plan, feature catalog, task DAG, SPEC/IMPL/VERIFY/OPS, PRs, audits, and post-merge verification.

LLM-authored goal text is draft evidence only. A Goal Contract becomes authoritative only when confirmed by the configured human or governance role.

### Trace To Goal
Every downstream artifact must be able to answer:

```text
Which Goal Contract sufficient condition does this satisfy?
What evidence proves that condition moved forward?
What remains uncovered?
```

A task or PR can be locally successful while still failing the goal trace. Shirube must report that distinction explicitly.

### Context Pack
A Context Pack is the reproducible input given to an AI coding agent for a task. It must include:

- task objective and task id;
- linked Goal Contract conditions;
- SSOT / SPEC / IMPL / VERIFY / OPS references;
- trace graph neighborhood;
- relevant files and docs;
- acceptance criteria;
- known blockers and escalation conditions;
- expected tests and gates;
- provider, permission, sandbox, and tool policy;
- output and evidence requirements.

Context Packs must be inspectable, hashable, and replayable. A worker prompt that only says "implement task X" is insufficient for strict/public workflows.

### AI Change Record
An AI Change Record is the structured evidence package for an AI-generated or AI-assisted PR. It must include:

- intent and task reference;
- Goal Contract and plan references;
- Context Pack id/hash;
- agent/provider identity and provider settings;
- permission mode, allowed tools, denied tools, MCP servers/tools, sandbox/network posture;
- changed files and major-change rationale;
- SPEC/IMPL/VERIFY/OPS trace status;
- impacted areas;
- tests run and results;
- acceptance criteria status;
- unresolved risks and escalation history;
- human approval / merge authority evidence;
- final merge decision reason.

The AI Change Record is the minimum artifact for enterprise auditability. A PR without it can be code-complete but cannot be enterprise-ready.

## 8. Evidence Model

```json

  "source_uri": "string|null",
  "artifact_path": "string|null",
  "artifact_hash": "sha256|null",
  "actor": {
    "type": "human|agent|github_user|system",
    "id": "string"
  },
  "summary": "string",
  "observed_at": "ISO-8601",
  "validity": "current|stale|superseded|invalid",
  "privacy_scope": "local|repo|public",
  "metadata": {}
}
```

Evidence requirements:
- append-friendly;
- stable artifact references or hashes when available;
- private reasoning traces are not evidence by default;
- secret-bearing or local-sensitive data is redacted before GitHub/public projection.

## 9. Gate Decision Model
All rule evaluations return:

```json
{
  "rule_id": "G2.hearing.required_confirmation",
  "gate": "hearing",
  "decision": "PASS|WARN|BLOCK|OBSERVE",
  "severity": "info|warning|error",
  "profile": "minimal|standard|strict",
  "message": "human-readable explanation",
  "evidence_refs": ["ev_..."],
  "remediation": "what to do next",
  "deterministic": true
}
```

The decision model is the shared contract for CLI, MCP, GitHub Actions, and local-only workflows.

Acceptance scenario:

```gherkin
Given a workflow action requires hearing evidence
When the Gate Engine evaluates the action for a strict profile
Then the decision includes PASS, WARN, BLOCK, or OBSERVE
And the decision includes rule id, profile, evidence refs, message, and remediation
And the decision does not depend on LLM natural-language judgment
```

## 11. Profiles
### minimal
- observe/warn by default;
- local-only friendly;
- no hard GitHub dependency;
- no public MCP-quality claim.

### standard
- GitHub may be task queue SSOT when configured;
- local `.framework/*` remains cache/draft in remote-enabled workflows;
- block remote publish and implementation split when required evidence is missing;
- warn on legacy/local paths during migration.

### strict
- block missing hearing, missing goal contract, missing role bindings, separation violations, failed validators, invalid phase transitions, missing goal trace, missing Context Pack, missing provider/tool policy, missing AI Change Record, missing audits, invalid release authority, and missing post-merge verification;
- concrete role bindings required;
- intended for public-grade dogfood and high-risk projects.

## 12. Control Mechanism Selection
Gate Engine decisions use deterministic TypeScript evaluators, CLI commands, and GitHub status/check outputs.

script 選定根拠: phase transitions, gate decisions, role readiness, publish readiness, and release authority must be reproducible without LLM judgment. TypeScript evaluators and CLI/GitHub checks provide deterministic inputs, explicit output JSON, and testable fixtures.

Hook 選定根拠: hooks are not canonical workflow state. Hooks may remain local immediate guards only for documented unavoidable cases such as pre-tool blocking, session context injection, state recovery, post-tool verification, and completion-time verification. They must not become the source of phase truth.

MCP 選定根拠: MCP tools are wrappers over the same state/rule model. MCP must not create a separate control plane or separate authority model.

## 15. Functional Requirements
- [SPEC-DOC4L-020-FR-001] PR A is docs-only and changes no runtime behavior.
- [SPEC-DOC4L-020-FR-002] PR B adds read-only `workflow-state/v1` semantics.
- [SPEC-DOC4L-020-FR-003] PR C exposes status/doctor/action-scoped check/explain observability without connecting new blocking behavior to existing workflow surfaces.
- [SPEC-DOC4L-020-FR-004] Local-only projects must work without GitHub.
- [SPEC-DOC4L-020-FR-005] `discover -> generate` remains compatible.
- [SPEC-DOC4L-020-FR-006] GitHub issue intake must not count as completed hearing.
- [SPEC-DOC4L-020-FR-007] Placeholder roles map to `setup_required`.
- [SPEC-DOC4L-020-FR-008] `draft_only` blocks remote publish actions.
- [SPEC-DOC4L-020-FR-009] No LLM output can approve phase completion or gate pass/fail.
- [SPEC-DOC4L-020-FR-010] A strict/public workflow must define a Goal Contract before phase planning, feature catalog approval, implementation split, or release claims.
- [SPEC-DOC4L-020-FR-011] Feature catalog entries, task DAG entries, SPEC/IMPL/VERIFY/OPS docs, PRs, audits, and post-merge verify results must trace to Goal Contract sufficient conditions.
- [SPEC-DOC4L-020-FR-012] Post-merge verification must update goal progress or explicitly report that no goal condition advanced.
- [SPEC-DOC4L-020-FR-013] A strict/public implementation start must have a Context Pack with stable id/hash and goal trace.
- [SPEC-DOC4L-020-FR-014] A strict/public implementation start must declare provider/tool policy, including permission mode, sandbox/network posture, and MCP/tool scope when applicable.
- [SPEC-DOC4L-020-FR-015] A strict/public PR must have an AI Change Record before release/merge readiness can pass.
- [SPEC-DOC4L-020-FR-016] Remote workflows must be able to project gate, trace, AI Change Record, and merge authority status into GitHub Checks or PR comments.
- [SPEC-DOC4L-020-FR-017] Agent-neutral evidence import must preserve provider identity and provenance without weakening gate semantics.
- [SPEC-DOC4L-020-FR-018] PRs that contribute to phase exit claims require post-merge verification evidence that names the PR, merge commit, verified ref, verifier/approver identity, checks run, verdict, residual risks, and explicit non-claims.

- Goal Contract, phase plan, feature catalog, task DAG, SPEC/IMPL readiness, Context Pack, provider/tool policy, AI Change Record, GitHub Check projection, audit, release, post-merge verify, and goal progress gates are specified and dogfooded;

- legacy `discover -> generate` compatibility is proven;
- status/check/explain output is stable for MCP/GitHub surfaces;
- #195 false-positive hardening is complete;
- security/privacy projection rules are documented;
- Shirube dogfoods the flow before Kodama/Totonoe broad rollout.
