# SPEC: Shirube Gate Engine

## 0. Meta
- Created: 2026-05-23
- Origin Issue: #197
- Implementation issue: #198
- Target release: v1.2.x public-grade Gate Engine baseline

## 1. Purpose
Define the public-grade Gate Engine architecture that unifies legacy discovery/generation, GitHub-backed workflow, role readiness, design gates, merge authority, MCP surfaces, and local-only operation.

This document is a specification baseline only. It introduces no runtime behavior change and must not make existing `shirube discover -> shirube generate` stricter.

## 2. Non-Goals
- Do not implement new blocking enforcement in this PR.
- Do not require GitHub for local-only projects.
- Do not create a second role or authority model.
- Do not replace `workflow-config.ts` or `merge-authority.ts`.
- Do not make hooks the canonical source of workflow state.
- Do not allow LLM output to decide phase completion, gate pass/fail, or exception approval.
- Do not apply this as a single oversized implementation PR.

## 3. Core Invariants
1. Shirube owns workflow state mechanically.
2. CLI, MCP, GitHub automation, and local-only usage consume the same state/rule model.
3. Phase transitions require deterministic evidence.
4. A GitHub Issue is intake evidence, not hearing completion.
5. Role readiness comes from `workflow-config.ts`.
6. Release and merge authority come from `merge-authority.ts`.
7. Every gate decision is deterministic, explainable, and evidence-backed.
8. Enforcement rolls out as observe -> warn -> block by profile.
9. Mutable facts have exactly one canonical owner; projections, caches, and snapshots are derived evidence/read models, not independent truth.
10. DB/state designs must describe primary keys, unique constraints, foreign keys or resolver boundaries before strict implementation gates pass.

## 4. Phase Model
Initial phase set:

```text
uninitialized
started
intake_ready
hearing_in_progress
hearing_blocked
hearing_complete
design_draft_ready
design_review
design_gate_passed
implementation_planned
implementation_in_progress
change_intake_required
review_ready
audit_ready
release_ready
released
archived
```

Each phase must define:
- allowed actions;
- blocked actions;
- required evidence;
- next legal transitions;
- remediation when blocked.

Backward movement is allowed only through explicit change/exception evidence.

## 6. Evidence Model
Evidence records must preserve provenance:

```json
{
  "id": "ev_...",
  "kind": "discovery_session|current_session|hearing_answer|human_confirmation|github_issue|design_artifact|role_binding|validator_result|review|audit|read_receipt|merge_authority|exception",
  "source_uri": "string|null",
  "artifact_path": "string|null",
  "artifact_hash": "sha256|null",
  "actor": {
    "type": "human|agent|github_user|system",
    "id": "string"
  },
  "summary": "string",
  "observed_at": "ISO-8601",
  "validity": "current|stale|superseded|invalid",
  "privacy_scope": "local|repo|public",
  "metadata": {}
}
```

Evidence requirements:
- append-friendly;
- stable artifact references or hashes when available;
- private reasoning traces are not evidence by default;
- secret-bearing or local-sensitive data is redacted before GitHub/public projection.

## 7. Gate Decision Model
All rule evaluations return:

```json
{
  "rule_id": "G2.hearing.required_confirmation",
  "gate": "hearing",
  "decision": "PASS|WARN|BLOCK|OBSERVE",
  "severity": "info|warning|error",
  "profile": "minimal|standard|strict",
  "message": "human-readable explanation",
  "evidence_refs": ["ev_..."],
  "remediation": "what to do next",
  "deterministic": true
}
```

The decision model is the shared contract for CLI, MCP, GitHub Actions, and local-only workflows.

Acceptance scenario:

```gherkin
Given a workflow action requires hearing evidence
When the Gate Engine evaluates the action for a strict profile
Then the decision includes PASS, WARN, BLOCK, or OBSERVE
And the decision includes rule id, profile, evidence refs, message, and remediation
And the decision does not depend on LLM natural-language judgment
```

## 8. Gate Catalog v0
| Gate | Purpose | Canonical source |
|------|---------|------------------|
| G0.start_boundary | Session, project identity, profile, activation readiness | current-session / local project metadata |
| G1.intake | Local request, MCP request, or GitHub issue captured | local request / GitHub issue |
| G2.hearing | Required questions, open questions, assumptions, human confirmation | discover session / hearing evidence |
| G3.role_readiness | Role binding, role separation, publish policy readiness | `workflow-config.ts` |
| G4.design_artifact | Design docs exist and link to intake/hearing evidence | docs / artifact hashes |
| G5.design_validation | Deterministic design validators | `gate design`, including LLM Control and Data Authority validators |
| G6.implementation_split | Issue/PR/task split derives from approved design state | local task output or GitHub Issues |
| G7.change_intake | Active-work design/scope change intake | change request evidence |
| G8.review_audit | Review, audit, read-receipt evidence | review/audit/read-receipt records |
| G9.release_authority | Release/merge authority | `merge-authority.ts` |

## 9. Profiles
### minimal
- observe/warn by default;
- local-only friendly;
- no hard GitHub dependency;
- no public MCP-quality claim.

### standard
- GitHub may be task queue SSOT when configured;
- local `.framework/*` remains cache/draft in remote-enabled workflows;
- block remote publish and implementation split when required evidence is missing;
- warn on legacy/local paths during migration.

### strict
- block missing hearing, missing role bindings, separation violations, failed validators, invalid phase transitions, and invalid release authority;
- concrete role bindings required;
- intended for public-grade dogfood and high-risk projects.

## 10. Control Mechanism Selection
Gate Engine decisions use deterministic TypeScript evaluators, CLI commands, and GitHub status/check outputs.

script 選定根拠: phase transitions, gate decisions, role readiness, publish readiness, and release authority must be reproducible without LLM judgment. TypeScript evaluators and CLI/GitHub checks provide deterministic inputs, explicit output JSON, and testable fixtures.

Hook 選定根拠: hooks are not canonical workflow state. Hooks may remain local immediate guards only for documented unavoidable cases such as pre-tool blocking, session context injection, state recovery, post-tool verification, and completion-time verification. They must not become the source of phase truth.

MCP 選定根拠: MCP tools are wrappers over the same state/rule model. MCP must not create a separate control plane or separate authority model.

## 13. Functional Requirements
- [SPEC-DOC4L-020-FR-001] PR A is docs-only and changes no runtime behavior.
- [SPEC-DOC4L-020-FR-002] PR B adds read-only `workflow-state/v1` semantics.

## 14. Public-Grade Readiness
The Gate Engine is not public-grade until:
- PR A/B/C are merged;
- strict-mode fixtures pass locally and in CI;
- legacy `discover -> generate` compatibility is proven;
- status/explain output is stable for MCP/GitHub surfaces;
- #195 false-positive hardening is complete;
- security/privacy projection rules are documented;
- Shirube dogfoods the flow before Kodama/Totonoe broad rollout.
