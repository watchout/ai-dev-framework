# SPEC-DOC4L-017 Operations (OPS)

> 4 layer template per ADF v1.2.4 (PR #133)、対応 SPEC: `docs/spec/v1.2.6-spec-audit-gate.md`、IMPL: `docs/spec/v1.2.6-impl.md`、VERIFY: `docs/spec/v1.2.6-verify.md`

## §1 概要

本 OPS doc は SPEC-DOC4L-017 (v1.2.6 Spec Audit Gate) の operational layer。SPEC ↔ IMPL ↔ VERIFY が「何を / どう作って / どう検証するか」を凍結するのに対し、本 OPS layer は「実環境でどう deploy / 監視 / 障害対応するか」を記録する。

SPEC-DOC4L-017 は **bundled CLI tool** 特性 (= `framework gate validate spec` サブコマンド、`src/cli/index.ts` 経由起動)。npm publish された stand-alone tool ではなく、ai-dev-framework repo 内蔵 tool として CI workflow + 開発者ローカルから直接実行される。したがって本 OPS は server-side service と異なり、deploy / rollback / monitoring が **minimal** で記述される。

## §2 Deploy

### §2.1 配布形態

- ai-dev-framework repo の `src/cli/` 配下に bundled、`package.json` の `"bin": {"framework": "./dist/cli/index.js"}` 経由起動
- 開発者ローカル: `npm run framework -- gate validate spec ...` または `tsx src/cli/index.ts gate validate spec ...`
- CI: `.github/workflows/spec-audit.yml` の step として `npx tsx src/cli/index.ts gate validate spec ...` を invoke

### §2.2 deploy event = git merge to main

- 本 CLI は server service ではないため、deploy event は **PR merge to main**
- merge により `src/cli/commands/gate/**` が main 反映 → 次 PR 以降の CI run + 開発者 local で新版が即時利用可能
- separate release artifact なし、npm publish なし (= internal tool)

### §2.3 dependency 配布

- 依存: `commander` (既存 dep)、`vitest` (devDep)、`tsx` (devDep)
- 新規 dep 追加: 本 PR シリーズで 0 件 (= 既存 dep のみ、`fetch` は Node 22+ native)
- Node version: 22.x (Workflow `actions/setup-node@v4` で固定)

## §3 Monitoring

### §3.1 primary signal: CI workflow status

- `.github/workflows/spec-audit.yml` の job (`validate`) の status check
- GitHub PR UI で各 PR ごと workflow run 結果 visible
- exit 2 → check fail → (v1.2.7 Task 1 で required check 化後) merge button disable

### §3.2 audit log

- production runner (CI / 開発者 local) で `.framework/audit/spec-audit-YYYY-MM-DD.jsonl` に追記
- jsonl entry schema: `{ts, opts, result_summary: {passed, exit_code, finding_count}, findings: [...]}`
- 1 entry = 1 CLI run、追跡時系列 audit possible

### §3.3 metric (= 集計対象、proposal)

- 各 PR の `spec-audit` run の `finding_count` を集計、月次で trend 可視化
- F1-F7 別 finding 件数の経時推移 = governance 健全性指標
- 集計 script は本 PR scope 外、follow-up issue 候補

## §4 Failure response

### §4.1 workflow FAIL 時の手順 (= PR author の対応)

1. PR の `spec-audit (SPEC-DOC4L-017)` workflow run logs を確認
2. step `Run validate-spec on real corpus` の stderr/stdout から findings list を取得
3. F1-F7 のうち何件 FAIL か特定 (= severity='fail' findings)
4. 該当 spec doc 修正 (= frontmatter / heading / §10 / §0 patch table 等)
5. push → workflow rerun → PASS 確認 → merge

### §4.2 false positive 疑い時

- spec doc が正しく、CLI logic が誤検知している疑い時:
- (a) Issue 起票、現 finding を再現 minimal corpus と共に記録
- (b) test fixture 追加 PR (= regression discipline § VERIFY §4 per)
- (c) 必要であれば adapter / check 修正 PR (= IMPL 改変、CI で 15 件 全 PASS が merge gate)

### §4.3 CLI logic bug 疑い時 (= 例: Task 3 meta_spec regex bug)

- VERIFY §5 Known gap として記録、hotfix PR で対応
- bug 期間中 affected spec は frontmatter side で workaround (= comment 削除等) 可能、IMPL fix を待たず unblock

### §4.4 CI workflow 自体の障害

- GitHub Actions infrastructure 障害時は GitHub status page で確認
- Node 22.x release breakage 時は `actions/setup-node@v4` の `node-version` pin で対応 (現状 22 floating major)

## §5 Rollback

### §5.1 CLI tool の rollback

- 静的解析 CLI のため、production への副作用 (= DB write / 外部 API call) なし
- bug 発生時は **forward fix** (= hotfix PR 起票 → merge) が primary、revert PR は exception
- 例外的 revert が必要な場合 (= 広範な false positive で全 PR block 等): `git revert <merge-commit>` PR、CTO L3 sanity 経由

### §5.2 branch protection rule の rollback

- v1.2.7 Task 1 で main branch に `validate` required check 登録予定
- rule 削除 (= governance loophole 復活) は CEO/CTO 判断 + admin action
- false positive 多発で開発 throughput が著しく低下時のみ検討、原則 forward fix
