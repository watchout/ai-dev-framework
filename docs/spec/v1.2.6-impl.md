## §2 アーキテクチャ

### §2.1 Adapter / Port 設計

外部世界アクセスはすべて 4 port のいずれか経由。`node:fs` / `node:child_process` / `fetch` / `node:https` を `validateSpec()` 本体および 7 check sub-module から **直接呼ぶことは禁止**。Node API 直 import / 呼出は adapter 実装層 (`src/cli/commands/gate/adapters/`) のみ。

| Port | Method | Adapter (production) | Adapter (test) |
|------|--------|----------------------|----------------|
| `SpecRepositoryPort` | `list()` / `parseFrontmatter(path)` / `sectionBody(path, heading)` | `adapters/spec-fs.ts` | fakes.makeFakeSpec |
| `GitHistoryPort` | `isPreExistingSpec(path, baseRef)` / `fileAtRef(path, ref)` | `adapters/git-history.ts` | fakes.makeFakeGit |
| `LinkProbePort` | `head(url, opts)` | `adapters/link-probe.ts` (`makeHttpLinkProbe`) | `makeFakeLinkProbe` |
| `AuditLogPort` | `append(entry)` | `adapters/audit-log.ts` (`makeFileAuditLog`) | fakes.makeFakeAudit |

port 抽象を維持することで、CI 環境 (`--link-probe=fake`) と production (`--link-probe=real`) で同 logic を再利用、test では fake で network 排除。

### §2.2 データフロー

```
CLI parser (cli.ts)
  → ValidateSpecOptions (raw string | string[])
  → validateSpec(opts, ports) [validate-spec.ts]
    → usage validation (strict×bootstrap mutex / unknown check / invalid linkProbe)
    → ports.spec.list() (port-failure → exit 2)
    → for each check in selected CheckName[]:
      → checks/<name>.ts (受け取り: CheckContext = {ports, baseRef, strict, bootstrap, linkProbe})
      → returns Finding[]
      → applySeverity(): --strict / bootstrap-downgrade per file
    → ports.audit.append({ts, opts, result_summary, findings})
    → returns ValidateSpecResult {passed, exit_code, findings, audit_log_path}
  → CLI prints findings → process.exitCode
```

## §3 主要 component (実装単位)

### §3.1 SpecFs adapter

`src/cli/commands/gate/adapters/spec-fs.ts`:

- `listMarkdown(root)`: 再帰的に `*.md` を列挙 (`node:fs/promises.readdir`)
- `parseFrontmatterImpl(content)`: YAML frontmatter から `id:` (single + list 両形式) と `meta_spec: true` flag を抽出
- `parseHeadingsImpl(content)`: ATX heading (`^#{1,6}\s+...$`) を全行列挙
- `sectionBodyImpl(content, heading)`: 指定 heading 直下から **同 or 上位 level heading** までを抽出 (= 親 section に含まれる sub heading は body に含む)

cache: 同 PR run 内で `readFile` 結果を Map で memoize、I/O 最小化 (§6.1 性能要件 per)。

### §3.2 AxisCheck (F1-F7)

`src/cli/commands/gate/checks/*.ts` — 各 check は 1 sub-module、引数 `CheckContext`、戻り値 `Finding[]`:

| FR | file | severity default | spec ref |
|----|------|-------------------|----------|
| FR-001 | `id-uniqueness.ts` | fail | SPEC §4.1 |
| FR-002 | `template-compliance.ts` | fail | SPEC §4.2 (+ F2-a/b/c/d sub-rule) |
| FR-003 | `one-id-per-file.ts` | fail | SPEC §4.3 |
| FR-004 | `control-mechanism.ts` | fail | SPEC §4.4 (script/Hook 選定根拠 + Hook 採用時 不可避 4 case) |
| FR-005 | `backward-compat.ts` | fail (bootstrap で warn / strict で block) | SPEC §4.5 (semantic 比較 + §0 patch table) |
| FR-006 | `notion-link.ts` | warn (strict で fail 昇格) | SPEC §4.6 (`[文献確認:...]` ラベル付帯) |
| FR-007 | `completion-literal.ts` | warn (strict で fail 昇格) | SPEC §4.7 (`程度|ぐらい|大体|状況に応じて`) |

F2 (template-compliance) は §0.1 / §4.2 per、frontmatter `metaSpec === true` ファイルを §11 のみで pass、§12-§14 欠如許容。F2-a (§7 受入基準 空白) → F2-b (Gherkin) → F2-c (testing layer 存在) → F2-d (layer keyword) の 4 sub-rule を順序付きで適用。

### §3.3 GateValidator

`src/cli/commands/gate/validate-spec.ts` — orchestrator。

- `SEVERITY` table: 各 CheckName に default severity / `--strict` で fail 昇格 / bootstrap 降格可否を記録
- `narrowChecks(raw)`: raw `string | string[]` から CheckName[] を narrow、未知 → usage error (no `as` cast)
- `narrowLinkProbe(raw)`: raw → `'real' | 'fake'` narrow、不正 → usage error
- `applySeverity(check, finding, ctx)`: 凍結 severity rule を適用、bootstrap downgrade は `shouldDowngradeBootstrap()` (= per-file `git.isPreExistingSpec` 判定) でファイル単位

### §3.4 CLI command

`src/cli/commands/gate/cli.ts`:

- `buildValidateSpecCommand()`: commander.Command を構築、`--check` / `--strict` / `--bootstrap` / `--base-ref` / `--link-probe` フラグを syntactic split のみで受領
- `buildValidateCommand()`: `validate` 親コマンドに `spec` を addCommand
- `gate.ts` 内で `gate.addCommand(buildValidateCommand())` 呼出、既存 `gate spec` (Gate 0) を破壊せず追加
