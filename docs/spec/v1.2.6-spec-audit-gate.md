## 5. インターフェース (Contract) [必須]

### 5.1 CLI 契約

```
framework gate validate spec [--check=<name>] [--strict]
  exit 0: 全 check pass
  exit 2: strict mode で 1+ fail
  --strict: warn を fail に昇格
  --check 指定なし: 全 7 check 実行

利用 example:
  framework gate validate spec --check=id-uniqueness
  framework gate validate spec --strict
```

### 5.2 CI gate 連携 (cycle 2 patch、Gap 2 解消、§5.3.1 と統一)

- `.github/workflows/spec-audit.yml` で本 CLI を **no-flag** で実行 (= §5.3.1 per-file 自動選別)
- exit 2 → PR merge block
- `--strict` / `--bootstrap` の force flag は migration / manual override 専用、CI default ではない
- **本 §5.2 は §5.3.1 を SSOT として参照**、二重化禁止 (cycle 1 で生じた `--strict` 固定 vs no-flag 自動選別の drift を本 patch で解消)

### 5.3 bootstrap (Phase 0 grandfather)
- 既存 spec は `--strict` ではなく `--bootstrap` mode で warn only
- 新規 spec (本 PR 以降) から `--strict` 適用
- 移行: 既 spec を順次 retrofit、各 retrofit PR で `--strict` 通過確認

### 5.3.1 bootstrap vs strict 適用判定規則 (cycle 2 patch、Gap 2 解消)

CI workflow / CLI 実装で「どの spec に bootstrap / strict を適用するか」 は以下 literal 規則で固定:

```
判定 logic (per spec file):
  let base_ref = <PR の base branch HEAD> (e.g., origin/main)
  let file_first_commit = git log --diff-filter=A --format=%H <base_ref> -- <file> | tail -1
  if file_first_commit が空 or PR diff 内で新規追加された file:
    → strict mode (本 PR 以降の新規 spec 扱い)
  else if file_first_commit が base_ref より古い:
    → bootstrap mode (既存 spec の grandfather 扱い、warn only)
  else:
    → strict mode (既 file の改訂、本 PR 以降の改訂は strict 適用)
```

**CLI option mapping**:
- `framework gate validate spec --strict`: 全 file を strict 強制 (CI workflow default ではない)
- `framework gate validate spec --bootstrap`: 全 file を bootstrap 強制 (= migration 期 force)
- `framework gate validate spec` (no flag): 上記 per-file judgement で自動選別

**CI workflow 契約**: `.github/workflows/spec-audit.yml` は **no flag** で実行、per-file 自動選別。drift 防止のため、workflow / spec body / 実装 CLI で本 §5.3.1 を SSOT として参照。
