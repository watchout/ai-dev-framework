# Gate Overview

Shirube uses two gate families. Gate A/B/C are CLI-enforced pre-code checks.
Gate 1/2/3 are role and skill-driven review gates used after design,
implementation, and release preparation. They are complementary, not
interchangeable.

## Gate Map

| Name | Alias | When | Enforced by | Auto / Manual |
|------|-------|------|-------------|---------------|
| Gate A | Gate 1 pre / environment readiness | Before code | CLI hook / `shirube gate check-a` | Automatic |
| Gate B | Pre-code planning gate | Before code | CLI hook / `shirube gate check-b` | Automatic |
| Gate C | Pre-code SSOT gate | Before code | CLI hook / `shirube gate check-c` | Automatic |
| Gate 1 | Design validation | After design | `/gate-design` skill or equivalent review role | Manual |
| Gate 2 | Quality sweep | After implementation | `/gate-quality` skill or equivalent review role | Manual |
| Gate 3 | Adversarial release review | Before release / merge | `/gate-release` skill or L3 review role | Manual |
| Gate D | Post-deploy verification | After deploy | TBD | Planned |

## How to Read Gate Names

- Use Gate A/B/C when discussing local or CI-enforced readiness before code
  starts.
- Use Gate 1/2/3 when discussing role-separated design, quality, and release
  review.
- Do not treat a passed Gate A/B/C result as approval for Gate 1/2/3.
- Do not treat a role review as a replacement for required CLI or CI checks.

## Completion Evidence

Completion evidence must come from the surface that owns the gate:

| Gate family | Evidence source |
|-------------|-----------------|
| Gate A/B/C | CLI output, CI check, or `.framework/gates.json` generated by `shirube gate check` |
| Gate 1/2/3 | PR review, PR comment, audit report, QA report, or CTO/L3 decision record |
| Gate D | Runtime smoke result, rollback/recovery evidence, or post-deploy verification record |
