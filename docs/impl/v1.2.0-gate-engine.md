## 2. Adapter Boundaries

### Discovery adapter
Maps legacy discovery artifacts into hearing evidence.

Rules:
- completed discovery session -> `hearing_complete` evidence;
- incomplete/missing discovery -> missing evidence;
- do not change `shirube generate` behavior in PR B/C.

### GitHub Issue adapter
Maps Issue body/comments into intake evidence.

Rules:
- Issue text may pre-fill answers;
- Issue text never completes hearing by itself;
- GitHub is optional for local-only projects.

### Workflow config adapter
Use existing functions from `workflow-config.ts`.

Rules:
- placeholder roles -> `setup_required`;
- role separation violations are represented as gate findings;
- publish policy maps to allowed/blocked actions.

### Merge authority adapter
Use `merge-authority.ts` as the only release/merge authority evaluator.

Rules:
- do not duplicate merge authority logic;
- expose merge authority as G9 decisions/evidence;
- adapter errors are represented as explainable findings.

## 3. Data Model Notes
PR B should prefer pure evaluators and deterministic readers. Network access belongs in explicit adapters and must be optional.

The initial state object may be synthesized instead of persisted. If persisted later, migration must preserve existing v1 current-session files.

Evidence ids may use deterministic hashes or generated `ev_*` ids. PR B must choose one and document it before writing evidence to disk.

## 4. No-Break Rules
- Do not break `shirube discover -> shirube generate`.
- Do not make GitHub mandatory.
- Do not make repository topic activation a hard blocker.
- Do not introduce a second role model.
- Do not make hooks canonical state.
- Do not allow LLM-generated text to become approval evidence.

## 5. Control Mechanism Selection
Gate Engine control uses deterministic TypeScript evaluators and CLI/GitHub check outputs.

Hook usage is not the canonical state authority. Hooks may remain compatibility surfaces or immediate local guards, but state truth must come from the workflow state/rule engine.

## 6. Rollout Sequence
1. PR A: docs-only baseline.
2. PR B: read-only state model and adapters.
3. PR C: status/doctor/action-scoped check/explain observability.
4. PR D: action-to-surface connection design and hearing/design bridge.
5. PR E: enforcement expansion.
6. PR F: MCP surface and dogfood readiness.

Do not introduce implicit blocking enforcement before PR A/B/C are complete. PR C may expose explicit `workflow check --action ...`, but no existing command, hook, CI, or GitHub automation may call it by default until a later PR maps actions to enforcement surfaces.
