## 2. Adapter Boundaries

### Goal Contract adapter
Maps project goal artifacts into lifecycle authority.

Rules:
- V0/V1 sufficient conditions are first-class workflow inputs;
- LLM-authored goal text is draft evidence only;
- approved goal evidence requires the configured human/governance owner;
- feature catalogs, task DAGs, SPEC/IMPL/VERIFY/OPS docs, PRs, audits, and post-merge verification reference goal condition ids;
- missing goal trace is reported separately from local implementation success.

### Context Pack adapter
Builds and reads reproducible task execution inputs.

Rules:
- context pack generation consumes Goal Contract, phase plan, feature catalog, task DAG, SPEC/IMPL/VERIFY/OPS docs, trace graph neighborhood, run state, blockers, tests, and merge policy;
- every context pack carries stable id/hash and target provider format (`claude`, `codex`, `cursor`, `copilot`, `generic`);
- context pack output is inspectable and replayable;
- strict/public implementation must not start from a generic one-line task prompt.

### Provider/tool policy adapter
Maps provider execution configuration into auditable evidence.

Rules:
- preserve provider identity, model/surface when known, permission mode, allowed tools, denied tools, sandbox posture, network posture, MCP server/tool scope, and worktree isolation;
- do not assume Claude/Codex semantics for generic providers;
- external/cloud agents may import equivalent evidence, but imported evidence must not bypass Shirube gates.

### AI Change Record adapter
Builds the PR-level evidence package for AI-assisted changes.

Rules:
- AI Change Record references task id, Goal Contract condition ids, Context Pack id/hash, provider/tool policy, changed files, trace status, tests, audits, unresolved risks, escalation history, and merge authority evidence;
- generated records are deterministic markdown/json artifacts;
- GitHub PR comments and Check Runs project summaries only; local/private artifacts remain governed by `privacy_scope`.

### Discovery adapter
Maps legacy discovery artifacts into hearing evidence.

Rules:
- completed discovery session -> `hearing_complete` evidence;
- incomplete/missing discovery -> missing evidence;
- do not change `shirube generate` behavior in PR B/C.

### GitHub Issue adapter
Maps Issue body/comments into intake evidence.

Rules:
- Issue text may pre-fill answers;
- Issue text never completes hearing by itself;
- Issue text never approves a Goal Contract by itself;
- GitHub is optional for local-only projects.

### Workflow config adapter
Use existing functions from `workflow-config.ts`.

Rules:
- placeholder roles -> `setup_required`;
- role separation violations are represented as gate findings;
- publish policy maps to allowed/blocked actions.

### Merge authority adapter
Use `merge-authority.ts` as the only release/merge authority evaluator.

Rules:
- do not duplicate merge authority logic;
- expose merge authority as G15 decisions/evidence;
- adapter errors are represented as explainable findings.

## 3. Data Model Notes
PR B should prefer pure evaluators and deterministic readers. Network access belongs in explicit adapters and must be optional.

The initial state object may be synthesized instead of persisted. If persisted later, migration must preserve existing v1 current-session files.

Evidence ids may use deterministic hashes or generated `ev_*` ids. PR B must choose one and document it before writing evidence to disk.

Goal condition ids must be stable within a Goal Contract revision. If a goal condition changes, downstream trace entries either migrate explicitly or become stale/superseded evidence.

Recommended initial types:

```ts
type GoalContractStatus = "missing" | "draft" | "approved" | "verified";

interface GoalCondition {
  id: string;
  description: string;
  requiredEvidence: string[];
  status: "uncovered" | "planned" | "in_progress" | "verified";
}

interface GoalTraceRef {
  goalId: string;
  conditionIds: string[];
  artifactPath?: string;
  artifactHash?: string;
}

interface ContextPackRef {
  id: string;
  taskId: string;
  providerTarget: "claude" | "codex" | "cursor" | "copilot" | "generic";
  artifactHash: string;
  goalTrace: GoalTraceRef;
}

interface AIChangeRecordRef {
  id: string;
  pullRequest?: number;
  taskId: string;
  contextPackId: string;
  artifactHash: string;
  status: "draft" | "complete" | "projected";
}
```

## 4. No-Break Rules
- Do not break `shirube discover -> shirube generate`.
- Do not make GitHub mandatory.
- Do not make repository topic activation a hard blocker.
- Do not introduce a second role model.
- Do not make hooks canonical state.
- Do not allow LLM-generated text to become approval evidence.

## 5. Control Mechanism Selection
Gate Engine control uses deterministic TypeScript evaluators and CLI/GitHub check outputs.

Hook usage is not the canonical state authority. Hooks may remain compatibility surfaces or immediate local guards, but state truth must come from the workflow state/rule engine.

## 6. Rollout Sequence
1. PR A: docs-only baseline.
2. PR B: read-only state model and adapters.
3. PR C: status/doctor/action-scoped check/explain observability.
4. PR D: Goal Contract, phase plan, hearing/design bridge, and action-to-surface connection design.
5. PR E: feature catalog, task DAG, SPEC/IMPL readiness, Context Pack, provider/tool policy, pre-impl audit, and implementation audit enforcement.
6. PR F: AI Change Record, GitHub Check projection, release authority, post-merge verify, goal progress review, MCP surface, and dogfood readiness.

Do not introduce implicit blocking enforcement before PR A/B/C are complete. PR C may expose explicit `workflow check --action ...`, but no existing command, hook, CI, or GitHub automation may call it by default until a later PR maps actions to enforcement surfaces.
