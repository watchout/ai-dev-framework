## 2. Core Evaluator
Create a pure evaluator in `src/cli/lib/merge-authority.ts`.

Inputs:
- framework config;
- PR metadata: number, head SHA, base branch, draft state, labels;
- review list: author login, state, commit SHA, submittedAt, dismissed state when available;
- optional team membership resolver result;
- workflow policy and audit level.

Output:

```ts
type MergeAuthorityDecision =
  | { status: "pass"; required: RequiredAuthorityEvidence[] }
  | { status: "block"; reason: string; missing: RequiredAuthorityEvidence[]; details: string[] };
```

The evaluator must not call GitHub directly. Fetching belongs to adapters so tests can cover all cases deterministically.

## 3. GitHub Adapter
`src/cli/lib/github-reviews.ts` fetches:
- current `headRefOid`;
- draft state;
- labels;
- reviews with reviewer login, review state, and commit id;
- team membership if `github_team` bindings are used.

The adapter must treat API errors as blocking in CI/check mode.

The adapter must expose enough review metadata for the evaluator to reduce each bound actor/team to its effective latest non-dismissed current-head review state.

## 4. Role Identity Resolver
Extend role binding support for merge authority identities:

```json
{
  "roles": {
    "bindings": {
      "l3_governance_owner": { "type": "github_user", "id": "cto-login" },
      "release_owner": { "type": "github_user", "id": "watchout" },
      "human_approver": { "type": "github_user", "id": "watchout" }
    }
  }
}
```

If the general role schema cannot add `github_user` / `github_team` in the MVP, add a dedicated `mergeAuthority.identities` mapping and keep `.roles.bindings` as the role SSOT. The implementation must still fail closed when no GitHub identity can be resolved.

## 5. Commands

### `shirube merge doctor`
Read-only command.

Responsibilities:
- fetch PR metadata and reviews;
- run the shared evaluator;
- print required evidence and blockers;
- exit 0 on pass, non-zero on block.

### `shirube merge`
Convenience command.

Responsibilities:
- run `shirube merge doctor` internally immediately before merge;
- refuse merge if doctor blocks;
- call GitHub merge only after doctor passes.

CLI is not sufficient enforcement. The required status check is mandatory.

### `shirube merge-authority`
CI/check command.

Responsibilities:
- run the same evaluator for the current PR/head;
- emit pass/fail status through GitHub Actions;
- produce machine-readable JSON output for audit logs.

## 6. Workflow
Add `merge-authority.yml`.

Triggers:
- `pull_request` on opened, synchronize, reopened, ready_for_review, labeled, unlabeled.
- `pull_request_review` on submitted, edited, dismissed.

Behavior:
- fail on draft PR when workflow policy is `draft_only`;
- evaluate current head SHA;
- reduce reviews by actor/team to the latest non-dismissed current-head state;
- fail closed on GitHub API or config errors.

Workflow triggers:
```yaml
on:
  pull_request:
    types: [opened, synchronize, reopened, ready_for_review, labeled, unlabeled]
  pull_request_review:
    types: [submitted, edited, dismissed]
```

Wire GitHub API/Actions workflow and status output.

Add operator-facing commands that share the same core evaluator.

## 8. Compatibility
- Existing `external:*` role bindings remain valid for orchestration but not for merge authority evidence.
- Repositories may continue design/prep work without branch protection, but cannot claim public MCP-quality strict merge governance.
- `auto-merge.yml` paths based only on labels such as `audit-passed` must be removed or hardened to require `shirube merge-authority`.
